## 0. Что уже есть (фундамент, не переделываем)
- `Tenant.site_config` (JSON) — секции, тексты, nav, menus, archetypes, gallery,
  font, hero_style и т.д. Нормализация — `apps/tenants/siteconfig.py::normalize`
  (переживает расширение схемы; реестры `SECTIONS`, `NAV_ITEMS`, `FONTS`, `TEXT_FIELDS`).
- Кабинет `/dashboard/site/` (`site_view`) — карточки: Homepage builder / Navigation /
  Section covers / Design; галерея шаблонов (`sitetemplates.apply_template`).
- `/dashboard/site/home/` (`home_builder_view`) — порядок/видимость блоков (числа+чекбоксы).
- `/dashboard/site/menu/` (`menu_builder_view`) — JS-билдер многоуровневого меню (S7).
- `/dashboard/site/sections/` (`sections_view`) — обложки разделов (интро+hero+галерея).
- **Live-preview-эндпоинты (ключевой задел):**
  - `site_preview` — iframe актуальной витрины + переключатель ширины Desktop/Tablet/Mobile.
  - `site_preview_draft` (POST) — принять черновик композиции главной → merge в
    `site_config` → сессия `site_preview_draft`; витрина `/?preview=1` рендерит черновик
    БЕЗ записи в БД.
  - `site_inline_edit` (POST) — сохранить одно текстовое поле (белый список `TEXT_FIELDS`),
    отредактированное на превью (contenteditable).

Вывод: схема данных, реестр секций и контур «черновик→preview→save» готовы. M20 —
это **UI поверх них** (drag-drop, палитра, инлайн-правка, панель свойств).

## 1. Цель M20
Владелец собирает главную и разделы витрины **визуально, в одном экране с live-preview**:
перетаскивает секции, правит тексты на месте, добавляет/убирает блоки из палитры,
меняет настройки секции в боковой панели — без числовых полей порядка и без отдельных форм.

Нефункциональные рамки: прогрессивное улучшение (без JS — текущие формы остаются
фолбэком), CWV не ломаем (без тяжёлых SPA-фреймворков — vanilla/htmx + Sortable-lite),
GDPR (никаких внешних CDN-шрифтов/трекинга), `prefers-reduced-motion`.

## 2. Разбивка (ветка→CI→чекпоинт на каждую)

### M20a — единый экран билдера + live-preview (каркас)
- Новый `/dashboard/site/builder/` — слева панель секций (список из `site_config.sections`),
  справа iframe `/?preview=1`. Изменения в панели → debounce POST в `site_preview_draft`
  → iframe reload (или postMessage). Кнопка «Speichern» → запись черновика в `site_config`.
- Переиспользует `site_preview_draft` (уже мержит sections + archetypes). Добавить запись
  «применить черновик в БД» (сейчас draft живёт только в сессии) — новый POST
  `site_draft_apply` (нормализует и сохраняет, как inline_edit).
- Без новых полей. Тесты: draft в сессии, apply пишет в site_config, мусорные секции игнор.

### M20b — drag-drop порядка секций
- Sortable-список секций (легкий vanilla-drag, без зависимостей или мелкая lib без CDN —
  завендорить в `static/src`). Порядок секции = индекс в массиве (уходим от числового
  `order`; `normalize` уже держит список — нужно, чтобы порядок брался из позиции, а не из
  числа). Тогл видимости — чекбокс на карточке секции.
- Drop → пересборка `sections` → `site_preview_draft` → preview. Сохранение — M20a apply.
- **Урок-риск:** `normalize` сейчас сортирует/дополняет секции; согласовать, чтобы порядок
  массива был источником истины (не перетасовывал). Тест: переставленный порядок
  переживает normalize и рендер.

### M20c — inline-правка текста на превью
- Превью в режиме билдера: ключевые тексты (`TEXT_FIELDS`: hero title/subtitle, about …)
  как `contenteditable`; blur → `site_inline_edit` (уже есть). Расширить `TEXT_FIELDS`
  на CTA/FAQ/process/team по мере надобности (белый список — защита).
- Подсветка редактируемых зон только в билдере (флаг в preview-контексте), на боевой
  витрине — нет.

### M20d — палитра секций + панель свойств
- «+ Block» открывает палитру доступных секций (из `SECTIONS`, ещё не добавленных) →
  добавление в `sections`. Удаление — на карточке.
- Боковая панель свойств выбранной секции: специфичные поля (hero: style/фото; cta:
  кнопка/ссылка; faq/testimonials: пары; gallery: загрузка). Переиспользует существующие
  редакторы из `site_view`/`sections_view`, вынесенные в партиалы/JSON-эндпоинты.

### M20e — медиа в билдере
- Загрузка фото (hero/gallery/team) прямо из панели свойств — reuse
  `catalog.images.save_product_image`/`delete_stored_image` (Pillow-валидация, как S3b/⑤b).
- Превью обновляется после загрузки.

### M20f — тема вживую + шаблоны
- Контролы шрифта (`FONTS`, системные стеки), акцент (`primary_color`), hero_style —
  с мгновенным preview. Применение шаблона (`sitetemplates`) из билдера с предупреждением
  «перезапишет раскладку, тексты владельца сохранятся».

## 3. Данные и совместимость
- **Никаких новых моделей/миграций** (всё в `site_config`). Каждый инкремент сохраняет
  обратную совместимость: легаси-конфиги без нового ключа → `normalize` даёт дефолт.
- Реестр секций `SECTIONS` — единый источник; новая секция → обновить хардкод-список в
  `apps/tenants/tests/test_siteconfig.py::test_normalize_empty_gives_defaults` (известный урок).
- После правок шаблонов с новыми классами — `npm run build:css` (CI гейтит свежесть).

## 4. Риски
- `normalize` как «умная» функция (досортировывает/дополняет) конфликтует с «порядок =
  позиция в массиве» (M20b). Решение: в M20b явно зафиксировать инвариант порядка + тест.
- Перф превью при частых POST — debounce + reload только iframe (не весь кабинет).
- Соблазн SPA — НЕ тянем React/Vue (CWV/простота); vanilla + htmx + мелкий sortable.

## 5. Definition of Done (M20)
Владелец на `/dashboard/site/builder/` перетаскивает секции, правит тексты на месте,
добавляет/удаляет блоки, грузит фото и меняет тему — с живым превью и сохранением в
`site_config`, без числовых полей и без регрессии для легаси-витрин. Старые формы
остаются как фолбэк без JS.
