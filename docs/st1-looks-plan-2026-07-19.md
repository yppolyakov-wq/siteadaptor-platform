# ST-1 «Каталог Look'ов» — план (2026-07-19)

Этап B1 ТЗ `next-gen-master-tz-2026-07-19.md §3` (концепт `studio-concept-2026-07-18.md`).
Look = целостный визуальный образ сайта, применяемый одним кликом: палитра+шрифт+
типографика+стиль карточек+вид шапки+hero-стиль+набор секций (+тёмный вариант).
Разведка — фоновый Explore 2026-07-19; вывод: ВСЯ нижняя инфраструктура готова
(`sitetemplates.apply_template` идемпотентен, все ключи вида существуют в normalize,
живой канал превью `site_preview_draft` уже принимает всю пачку) — расширяем верх.

## 1. Решения

- **Реестр: 3 Look-СЕМЕЙСТВА × 14 архетипных палитр = 42 Look'а** (а не 42
  независимых рукописных пресета — сопровождаемость). Семейства (рабочие имена):
  **«Klar»** (светлый, system-шрифт, плоские карточки, nav classic),
  **«Warm»** (serif/rounded, мягкие тени, скругления, nav centered),
  **«Nacht»** (ТЁМНЫЙ, жирные заголовки, nav minimal) — Р4 «вкл. тёмные» закрыт.
  Per-архетип: свой accent на каждое семейство (`ARCHETYPE_LOOK_ACCENTS`) +
  набор секций из существующего рекомендованного TEMPLATE архетипа. Реестр —
  чистые данные в `apps/tenants/sitetemplates.py` (рядом с TEMPLATES), без БД.
- **Тёмный Look — новый site_config-ключ `theme`** (presence-minimal: ключ
  ТОЛЬКО при `"dark"` → golden целы, паттерн presence/classic_ui). Рендер:
  `_base.html` ставит `.dark`-класс по умолчанию при `theme=dark`, если
  пользователь не выбрал светлую вручную (localStorage-тумблер `sf-theme`
  остаётся сильнее — уважение выбора посетителя). Карта тёмных цветов в
  app.css уже есть (`.dark`).
- **Применение — расширить `apply_template` → `apply_look`**: к нынешним
  sections/texts/hero_style/accent добавить материализацию `font`/`typography`/
  `site_defaults`/`nav.style`/`theme`. Инварианты: тексты владельца НЕ
  затираются (как сейчас), фото владельца НЕ трогаем (фото-настроение — забота
  демо-китов), условные ключи (classic_ui/board/seo/presence/
  service_index_layout) НЕ материализуются, W6 — не создаём второй источник
  темы: галерея пишет ТОЛЬКО через apply_look/draft-канал.
- **Живая галерея**: мини-превью = scaled-iframe СВОЕГО сайта с Look'ом.
  Общий `session["site_preview_draft"]` — один слот, N превью затёрли бы друг
  друга → **stateless-параметр `?look=<family>`** на витрине под `?preview=1`
  (оверлей ключей Look'а поверх конфига в контекст-процессоре, только preview,
  ничего не пишет). Ленивая загрузка IntersectionObserver. Клик по карточке в
  билдере — существующий draft-канал (payload уже несёт всю пачку) → канва
  перекрашивается БЕЗ перезагрузки, Undo — существующий стек билдера.
- **classic_ui (железное правило §8b)**: новая галерея — только в новом виде;
  classic → прежняя галерея шаблонов; сам `apply_look` работает в обоих.

## 2. Инкременты

1. **ST-1a Движок**: ключ `theme` (normalize presence-minimal + `.dark`-дефолт
   в `_base.html`) · реестр `LOOK_FAMILIES`+`ARCHETYPE_LOOK_ACCENTS` ·
   `looks_for(business_type)` (3 карточки, свой архетип первым) · `apply_look`
   (поверх apply_template) · **адверсариальный замок** (образец
   CBLOCK_VARIANTS-замка): все 42 Look'а × apply→normalize без потерь +
   идемпотентность + golden целы + у каждого архетипа ровно 3.
2. **ST-1b Галерея**: preview-оверлей `?look=` · слайд мастера `stil` → «Look»
   (гейт classic_ui; scaled-iframe карточки) · область Look в билдере Site
   (клик → draft-канал, Apply → apply_look) · стенд-проверка «клик
   перекрашивает канву без перезагрузки, Undo жив».
3. **Финал**: i18n (лейблы семейств), CSS при новых утилитах, build-log/
   CLAUDE.md/ТЗ-статус, CI → FF-merge.

## 3. Замки/приёмка ТЗ

Пекарь видит 3 пекарских Look'а первыми ✅ (looks_for по business_type);
клик перекрашивает превью без перезагрузки ✅ (draft-канал+двойная буферизация);
Undo ✅ (стек билдера); каждый Look проходит apply+normalize ✅ (замок);
golden целы ✅ (apply — действие владельца; theme-ключ presence-minimal).
