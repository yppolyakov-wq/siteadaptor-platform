## 0. Контекст (зачем и почему сейчас)
- **Проблема.** Владелец ведёт входящие транзакции в 6+ разрозненных кабинет-экранах
  (заказы / брони / номера / билеты / Aufträge / резервы) — каждый плоский список или календарь со
  своими кнопками статуса. Нет единой «доски дел». Склад — голый счётчик `Product.stock_quantity`,
  перезаписывается на месте (`-= qty`), история движений теряется. Это последний крупный
  «лёгкий-ERP»-шаг мастер-трека (`unified-sellable-entity-master-track §0`: «заказ тоже
  унифицируется → единая Kanban-доска управления + управление остатками/склад»).
- **Де-риск (что уже готово).** **U-A/U-B/U-C закрыты.** Прецедент адаптера `SellableEntity`
  (`apps/core/sellable.py`) — тот же lazy-`get_model` подход применяем к транзакциям, **модели НЕ
  сливаем**. **E-7** уже отгрузил `Order.payment_method` (миграция `orders/0012`) → UD1-1 только
  ВЫСТАВЛЯЕТ поле в контракте (без миграции). `finance.RevenueEntry` — де-факто уже единый «сток»
  выручки по `(source, source_ref)`; `promotions.Customer` — единая личность клиента;
  `core.fsm.StateMachine.apply()`+`allowed_targets()` — единый контракт смены статуса.
- **Решения владельца (2026-07-07).** Kanban = **drag-drop колонки** (Trello-style, как в
  конструкторе сайта). Объём первой сессии = **вся волна U-D1..U-D3** (проекция + доска +
  склад-леджер, включая миграцию `StockMovement`). UD4 — следующей сессией.
- **Ранее зафиксировано** (`unified-sellable-entity-decisions-2026-06-30`): **D1** склад-леджер =
  append-only РЯДОМ со счётчиком (счётчик — истина, леджер — реконсиляция-вью); **D2** = per-app
  страницы ОСТАЮТСЯ + общая доска (доска — дополнение, не замена); **D3** SMS **ОТЛОЖЕН** (UD4-1
  снят из волны, поднять с WhatsApp позже); **D4** = line-items НЕ унифицируем (проекция отдаёт
  `title` + готовый `subtotal_display`, без конфликта OrderItem-int / JobLine-Decimal).
- **Итог первой сессии.** Единая проекция транзакций → живая drag-drop доска `/dashboard/board/`
  по всем sell-модулям → append-only склад-леджер `StockMovement` с реконсиляцией. ЛК и per-app
  экраны сохраняются (паритет).

## 1. Область и границы

**6 FSM-транзакций** (все `TimestampedModel` UUID-PK + `customer`→`promotions.Customer` PROTECT +
unique `reference_code` с 1-буквенным префиксом + свой `StateMachine`-подкласс):

| kind | Модель | Файл | Префикс | FSM | «Готово»-терминал |
|---|---|---|:--:|---|---|
| order | `orders.Order` | `apps/orders/models.py:18` | O- | `OrderSM` (`apps/orders/state_machine.py:11`) | picked_up / shipped |
| booking | `booking.Booking` | `apps/booking/models.py:307` | T- | `BookingSM` (`apps/booking/state_machine.py:11`) | fulfilled |
| stay | `stays.StayBooking` | `apps/stays/models.py:245` | S- | `StayBookingSM` (`apps/stays/state_machine.py:12`) | fulfilled |
| ticket | `events.Ticket` | `apps/events/models.py:346` | E- | `TicketSM` (`apps/events/state_machine.py:36`) | attended |
| job | `jobs.Job` | `apps/jobs/models.py:19` | A- | `JobSM` (`apps/jobs/state_machine.py:12`) | done→invoiced |
| reservation | `promotions.Reservation` | `apps/promotions/models.py:334` | R- | `ReservationSM` (`apps/promotions/state_machine.py:47`) | fulfilled |

`loyalty.GiftVoucher` — 7-я «транзакция», но БЕЗ `StateMachine` (`payment_state` меняется напрямую
`gift.mark_gift_voucher_paid`) → в v1 доски/леджера **не участвует**.

**Общая инфраструктура (уже есть, переиспользуем):** `promotions.Customer` (`apps/promotions/models.py:21`)
— единственная кросс-доменная личность (FK PROTECT из всех транзакций, кроме GiftVoucher).
`finance.RevenueEntry` (`apps/finance/models.py:18`) — append-only ledger выручки, дедуп по
`(source, source_ref)`, куда все FSM пишут на завершении → образец идемпотентности для склад-леджера.

### Партиция (что унифицируем / плагин / раздельно)
- **УНИФИЦИРУЕМ (одна реализация на все kind):** протокол `Transaction` (адаптер, модели не
  сливаются) · pipeline-маппинг `stage ∈ {intake, in_progress, done, terminal}` · drag-drop
  `_kanban_board.html` + generic `kanban_action` · партиал `_status_actions.html` из
  `allowed_targets` · один append-only склад-леджер `StockMovement` (движения orders+jobs).
- **ПЛАГИН (per-kind за единым интерфейсом):** статус→колонка · форма line-item (OrderItem int /
  JobLine Decimal / embedded rooms|quantity / slot без позиций) · шаблон уведомления ·
  availability-движок (за buy-box, U-D не трогает).
- **РАЗДЕЛЬНО (НЕ трогаем — движки):** 5 анти-оверселл атомарных путей · availability-математика ·
  `finance.record_revenue`/`record_reversal` на FSM-хуках · пер-app booking-календари ·
  `_restore_stock`/`commit_stock`/`F()`-side-effects (леджер их **логирует**, не заменяет).

### Ключевые инварианты (correctness-critical)
1. **Проекция/доска ЧИТАЮТ `status`; пишут статус только через `SM().apply()`.** ⚠️ Есть 3 боевых
   обхода FSM прямым `.status=` (`events/public_views.py:624`, `events/views.py:321` и `:309`) —
   read-проекция от них не зависит; `kanban_action` НЕ строит инвариант «status всегда валиден по
   FSM-переходам» (риск R1). Опц. вычистить 3 обхода — не входит в критпуть.
2. **Все деньги/склад-side-effects — на `on_transition`-хуках 6 FSM.** Доска зовёт тот же
   `apply()` → никакой параллельной логики переходов. `fsm.py:44` `src==dst` → no-op (перерисовка/
   повтор не задваивает выручку).
3. **Склад-счётчик реально двигают ТОЛЬКО 2 из 5** анти-оверселл движков: `orders`
   (`services.py:36` sale / `state_machine.py:73` restore) + `jobs` (`services.py:111` commit).
   `promotions` = лимит акции (`available_quantity`, не инвентарь); `stays`/`events`/`booking` =
   счёт занятости vs капасити → **вне складского леджера** (их истина — свои движки).

### UD2-2 — drag-drop Kanban-доска + generic `kanban_action` · без миграции · **[DRAG-DROP]**
`templates/core/_kanban_board.html` — колонки из `pipeline_for(kind)`, карточки из `Transaction`.
**Взаимодействие — native HTML5 drag-drop** (переиспользовать паттерн из конструктора:
`movePageBlock`/`data-*`-атрибуты в `templates/tenant/site_home.html`, **БЕЗ новой JS-зависимости** —
дом. стиль self-contained). На drop:
- клиент **оптимистично** переносит карточку в целевую колонку, шлёт POST `kanban_action(kind, pk)`
  с целевым статусом (маппинг колонка→dst-статус; если в колонке несколько статусов — целевой =
  первый допустимый из `allowed_targets`);
- сервер `apps/core/views.py::kanban_action` резолвит модель+SM и зовёт
  **`SM().apply(instance, dst, actor=user)`** (тот же путь, что KDS — revenue/notif/stock на хуках,
  **НЕ дублируются**); `IllegalTransition` → **HTTP 409** → клиент **откатывает** карточку в исходную
  колонку (snap-back) + тост «Übergang nicht möglich»;
- на карточке также рендерим кнопки-переходы (`_status_actions.html`) — доступность / мобайл /
  тач-фолбэк, когда drag неудобен.
**KDS переводится на новую доску** (kind=order) — паритет обязателен (живая gastro-поверхность
`apps/orders/views.py:82-133` + `templates/orders/_kitchen_board.html`).
- **Критерии:** доска рендерит колонки/карточки для любого kind; статус двигается **только через
  `SM().apply`**; **`RevenueEntry.count()` не растёт** от перерисовки/повтора; KDS-поллинг/HTMX
  паритетны; нелегальный drop откатывается по 409.
- **Тесты:** `apps/core/tests/test_kanban.py` (новый) + orders KDS-паритет. ⚠️ зона регрессии —
  снапшот-паритет KDS + гейт «apply один раз».

### UD3-1 — модель `StockMovement` + миграция + сервис · ⚠️ **МИГРАЦИЯ**
Новый TENANT-app **`apps/inventory`** (чище, чем в `catalog`). `StockMovement(TimestampedModel)`:
`product` FK / `variant` FK(null) / `kind ∈ {receipt, sale, adjustment, return, stocktake, commit}` /
`delta` int (знаковый) / `source` / `source_ref` / `note` / `actor`(null) / индексы
`(product, created_at)`, `(source, source_ref)`. Сервис `record_movement(...)` — **идемпотентный по
`(source, source_ref, kind)`** для событийных движений (`get_or_create` + UNIQUE, образец
`finance.record_revenue`), свободный для ручных. **НЕ трогает счётчик** в v1 (D1: append-only
alongside). Регистрация в `TENANT_APPS` (`config/settings/base.py`).
- **Критерии:** таблица создана схемной миграцией (TENANT-схема); `record_movement` идемпотентен по
  `source_ref` для событийных kind (дубль→no-op); ручные — без дедупа; **счётчик не меняется**.
- **Тесты:** `apps/inventory/tests/test_movement.py` (новый); идемпотентность по образцу `finance/tests`.
- ⚠️ **Единственная обязательная схемная миграция волны** — деплой на tenant-схемах владельцем
  (`./scripts/deploy.sh single`); локально `--create-db`.

### UD3-2 — врезка логирования в orders+jobs · без миграции · **correctness-critical**
`record_movement(...)` **рядом** со списанием/возвратом **только**: `apps/orders/services.py`
(kind=sale) + `apps/orders/state_machine.py::_restore_stock` (kind=return); `apps/jobs/services.py::commit_stock`
(kind=commit). **promotions / stays / events / booking — НЕ логируем.** Movement — в **той же
`@transaction.atomic`**, что и декремент, **НЕ заменяет** атомарный `UPDATE`/`select_for_update`.
- **Критерии:** каждое списание/возврат catalog-товара пишет movement в той же транзакции;
  **атомарность декремента цела**; `sum(delta)` по товару = изменение счётчика (реконсиляция — UD3-3).
- **Тесты:** параллельный анти-оверселл тест orders (`TransactionTestCase`+`ThreadPoolExecutor`,
  DoD `docs/references/patterns/anti-oversell.md`) **остаётся зелёным**; `test_movement` delta↔счётчик.

### UD3-3 — кабинет склада `/dashboard/stock/` + реконсиляция · возможна миграция
Приёмки (receipt +N), корректировки (adjustment ±N), low-stock (`stock_quantity ≤ threshold`),
инвентаризация (stocktake→adjustment на разницу). **Ручные приёмки/корректировки — единственный путь,
где movement меняет счётчик** (в одной транзакции). Реконсиляция: `sum(delta)` vs `stock_quantity`
(алерт при расхождении). Опц. `Product.low_stock_threshold` (миграция) ∥ tenant-глобальный порог
(без миграции). Файлы: `config/urls_tenant.py`, `apps/inventory/views.py`,
`templates/inventory/stock.html`, `apps/inventory/services.py`, опц. `apps/catalog/models.py`.

## 3. Критический путь и последовательность (первая сессия = U-D1..U-D3)
```
UD1-1 ─┬─ UD1-2 (ЛК)
       ├─ UD1-3 ───────────────── UD2-3
       └─ UD2-1 → UD2-2 → UD2-3
UD3-1 → UD3-2 → UD3-3        (параллельная ветка — новый app apps/inventory)
```
Критпуть: **UD1-1 → UD2-1 → UD2-2 → UD2-3**. UD3-* — независимая ветка (только UD3-2 трогает горячий
путь orders/jobs → гейт параллельным тестом). **Старт — UD1-1** (де-рискует всё, окупается в ЛК UD1-2).
Батч-режим: связные срезы подряд, локальный гейт (ruff + pytest затронутых), пуш стопкой → один CI.

## 5. Риски и гейты
1. **FSM не исключителен (R1):** `kanban_action` только через `SM().apply()`; illegal→409→snap-back;
   не строить инвариант «status всегда валиден по FSM».
2. **Не задвоить выручку:** доска зовёт `apply()`; `src==dst` no-op; дедуп `(source, source_ref)` —
   доска пишет ту же сущность тем же ключом. Гейт: перерисовка не растит `RevenueEntry`.
3. **Не сломать 5 анти-оверселл:** UD3-2 добавляет movement РЯДОМ (та же atomic), только orders/jobs.
   Гейт: параллельные тесты зелёные.
