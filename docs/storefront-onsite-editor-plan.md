# On-canvas редактор на витрине — план и подзадачи

> Запрос владельца (2026-06-27): редактировать сайт **прямо на витрине** — кнопкой
> «Изменить дизайн», тулбары/инспектор блоков на канве, правка любой страницы,
> переиспользуемые блоки и микрошаблоны, **левая** панель «как в фотокниге» (пресеты+
> шрифты), переключение устройств и пер-девайс правка, публикация с предпросмотром.
> Выбранный объём старта: **Стадия 1+2** (редактор на витрине + инспектор блока).

## Принципы
- **Только владелец** (Django-аутентификация на витрине = владелец; клиенты — отдельный
  `apps.account`). Публичная витрина для посетителей не меняется и не нагружается.
- **Черновик → публикация.** Правки копятся в черновике (Redis-сессия, эфемерно, TTL),
  витрина рендерит его по `?preview=1`; «Опубликовать» = одна запись JSON в `site_config`.
  «Отмена» сбрасывает черновик.
- **Переиспользуем существующее**: home_builder + live-iframe, `site_preview_draft`,
  обёртки `data-sf-section`, реестр секций/C-блоков, пресеты раскладки, `sitetemplates`,
  пары шрифтов, переключатель устройств, per-page layouts. Бэкенд по максимуму не трогаем.
- **Scope настроек:** глобальные (шрифт/акцент/шапка/подвал/стиль карточек) → «весь сайт»;
  по-страничные/по-блочные (раскладка/число/тексты) → «эта страница / выбранные».

## Нагрузка (оценка)
Посетители — 0 доп. нагрузки (тот же статичный JSON-рендер). Редактирование редкое,
«всплесками»: черновик в Redis, пере-рендер превью с debounce, публикация — 1 запись.
Опц. кэш витрины (`pagecache`) со сбросом при публикации. Масштабируется на тысячи тенантов.

## Режимы инспектора: «Обычный» и «Эксперт» (progressive disclosure)
Как в редакторах фотокниг — два уровня в левой панели, переключатель вверху:
- **Обычный** — несколько дружелюбных контролов на блок: *скруглить углы* (radius фото/
  баннера/карточек), число/колонки крупными кнопками, скрыть, переместить, сменить
  микрошаблон. Цель — «ребёнок соберёт».
- **Эксперт** — раскрывает всё: источник данных, точные отступы/радиусы/тени,
  типографика, per-devais оверрайды, scope «весь сайт/страница». Выбор режима
  запоминается (localStorage).
Новые **визуальные параметры блока** (radius/тень/фон) — это и есть «простые» контролы;
хранятся в `site_config` рядом с layout, применяются в рендере секций/карточек.

#### Дизайн SE-2d (по разведке 2026-06-28 — Вариант A, мастер-настройка «на весь сайт»)
**Решение владельца:** Вариант A — настоящий глобальный дефолт стиля с наследованием +
переопределением на странице/блоке. «Выбранные» в SE-2d не делаем (только «эта страница /
весь сайт» + лёгкая кнопка «применить раскладку ко всем лендингам разом»); полный мультивыбор
страниц — отдельной задачей позже.

**Что показала карта рендера (что переиспользуем, НЕ переписываем):**
- Раскладка сеток УЖЕ механизирована: `normalize_layout` + `grid_class_string`; 4 пер-страничных
  ключа (`catalog_layout`/`events_index_layout`/`stay_index_layout`/`detail_related_layout`) и 11
  grid-секций главной (`sections[].layout`) идут через ОДИН механизм.
- Визуальный стиль карточек `visual={radius,shadow}` (`section_visual`, `siteconfig.py:291-303`)
  читается ТОЛЬКО в `home.html` → CSS-переменные `--sf-r`/`--sf-sh` на обёртке секции; CSS в
  `_base.html` `[style*="--sf-r"] .sf-card{…}`. **Главный кандидат на глобальный дефолт.**
- Глобальный дизайн (font/accent/hero) уже «весь сайт» через `context.py:modules_nav` →
  CSS-переменные в `_base.html`. SE-2d ложится по тому же образцу.

**Точки рендера (файл:строка → ключ → страница):** главная `templatetags/siteui.py:94-101`
(`grid_classes`), visual в `home.html:14-15`; каталог `promotions/public_views.py:322`
(`catalog_layout`); товар `…:356-357` (`detail_related_layout`); события
`events/public_views.py:79-81` (`events_index_layout`); номера `stays/public_views.py:161`
(`stay_index_layout`). `.sf-card` — во всех `_*_card`/секционных партиалах.

**Особые/рискованные места (НЕ ломать):** `stay_index_layout` жёсткий `"mobile":1`;
`events_index_layout` дефолт `"list"`; preview-режим разбросан по public_views (каждый сам
читает `site_preview_draft`); `detail_related_layout` — только на детальной товара.

**Структура (минимальный Вариант A — стиль карточек глобально):**
```python
site_defaults = {"card_radius": int 0..24, "card_shadow": bool}  # 0/false = текущее поведение
```
Резолвер `effective_card_visual(config, key)` = `section_visual(config,key)` если у секции/страницы
задан override (radius>0 или shadow), иначе `site_defaults.card_*`. Рендер: глобальный `--sf-r`/
`--sf-sh` на обёртке витрины (`_base.html`) из `site_defaults`; пер-секционные inline-переменные
ПЕРЕОПРЕДЕЛЯЮТ (CSS-specificity уже так работает). Дефолты 0/false → пустой `site_defaults` =
текущее поведение (без регрессии для legacy-конфигов).

**Разбивка (CI-зелёные шаги):**
- **SE-2d-1** `siteconfig`: `site_defaults` в `normalize` (дефолты 0/false) + резолвер
  `effective_card_visual` + unit-тесты (legacy: пустой / только-глобальный / только-секционный
  override; победа override над глобальным).
- **SE-2d-2** Рендер: глобальный `--sf-r`/`--sf-sh` на обёртке витрины из `site_defaults`;
  пер-секционные остаются override. Тест: глобальный radius виден на каталоге/событиях/номерах,
  секционный побеждает.
- **SE-2d-3** UI: scope-переключатель «🌐 весь сайт / 📄 эта страница» в инспекторе; «весь сайт»
  пишет в `site_defaults` (card_radius/shadow), «эта страница/блок» — в существующий `visual`.
  `collect()`/`site_preview_draft`/`home_builder` POST + live-preview. Тесты GET/POST.
- **SE-2d-4 (опц.)** Кнопка «применить раскладку ко всем лендингам разом» (каталог+события+номера)
  — фронт, пишет 3 ключа одним пресетом.
- **SE-2d-5 (опц., чинит найденный гэп)** `site_preview_draft` НЕ читает `catalog_layout`/
  `events_index_layout`/`stay_index_layout` из payload (`collect` шлёт, хендлер игнорит) →
  live-preview раскладки лендинга не работает до Save. Добавить разбор (whitelisted preset).

#### Дизайн SE-3 (по разведке 2026-06-28 — порядок риск↑, миграций нет; карта — scratchpad se3-plan-skeleton.md)
Порядок реализации: **SE-3d (фон/отступы) → SE-3b глоб. (размер/начертание) → SE-3a (микрошаблоны)
→ SE-3c (пер-девайс, последним)**. Всё в JSON `site_config`; пустые значения = текущий вид (без
регрессии для legacy). Переиспользуем механизм `visual`/`site_defaults`/`effective_card_visual`

**SE-3d (фон/отступы карточек) — наименьший риск, первым.** Расширяем `visual={radius,shadow}` →
