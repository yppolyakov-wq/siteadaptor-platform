# Hotel / Pension (A5 «Übernachtung») — план наполнения архетипа

Статус: 🟡 предложение (2026-06-22). Конвенция: docs до кода → разбивка владельцу →
реализация по подзадачам (ветка `claude/hotel-…` → CI зелёный → чекпоинт).

Архетип A5 (date-range размещение) уже на движке `apps.stays` (Track E, ~80 %).
Этот док — карта добивания до «полного отеля» по присланному ТЗ модуля
онлайн-бронирования, с приоритетом под **наш сегмент: маленький отель DACH**
(Pension, B&B, Ferienwohnung, kleines Hotel — 1–20 номеров).

## 0. Принцип scope (что мы делаем и чего НЕ делаем)

ТЗ описывает спектр от «визитки маленькой Pension» до «большой сети с PMS». Наш
продукт — прямое бронирование для **малого** отеля. ТЗ §5.2 само относит к
«не обязательно для маленького отеля»: сложный Revenue Management, динамическое
ценообразование, сложная CRM/корп-тарифы, большой PMS. Берём это как ориентир.

**В scope (этот план):** богатые тарифы, питание, спецпредложения, промокоды,
быстрый поиск по датам с главной, дети, богатая карточка номера, условия отмены
видимые до оплаты, SEO-разметка отеля, Hausordnung, loyalty-тариф, рассылка гостям.

**Вне scope (заметка в roadmap, не делаем сейчас):**
- Метапоиск / Google Hotel Ads, полноценный Channel Manager (двусторонний),
  интеграция с большим PMS/замками/кассой — тяжёлая интеграционная инфраструктура;
  частично уже закрыто односторонним iCal (импорт занятости Booking/Airbnb + экспорт
  нашего фида). Полный CM — отдельный крупный модуль, не для MVP малого отеля.
- Роли сотрудников (Housekeeping/Revenue Manager/…), online-чекин, электронная
  регистрация гостей — для большой гостиницы (ТЗ §6). У нас уже есть `core.Membership`
  под роли — задел, но наполнение откладываем.
- Группы/корпоративные тарифы/агентские брони — редки у малого отеля; mini-вариант
  (заявка «для бизнеса») закрываем формой, без отдельного движка.

## 1. Что уже есть (фундамент, не переделываем)

- `StayUnit` (тип/quantity/цена·ночь/выходная цена/`SeasonRate`/min_nights/max_guests/
  deposit/manual-confirm/фото), `UnitBlock`, `ICalSource`, `StayBooking` (снимок цены,
  total, extras, deposit, payment_state, invoice_id).
- Анти-овербукинг по ночам (`services.book_stay`, `availability.range_available`).
- Витрина `/unterkunft/` (поиск по диапазону **в карточке номера**) → `/s/<code>/`.
- Extras (`core.Extra`, per-night) добавляются в брони на витрине и в total.
- Депозит онлайн (Stripe Connect), письма (created/confirmed/cancelled/reminder/owner),
  авто-Rechnung (`stay_to_invoice`, 7 % Beherbergung), iCal импорт/экспорт.
- Кабинет `/dashboard/stays/`: календарь загрузки, юниты, сезонные тарифы, блоки, iCal.
- Demo-kit `hotel` («Pension Seeblick»): 4 типа номеров, extras, season, deposit.

## 2. Гэпы по ТЗ → приоритизированные подзадачи

Приоритет = (леверидж для прямого бронирования малого отеля) × (близость к ТЗ-ядру
«даты/номера/тарифы/услуги/оплата/спецпредложения»).

### 🥇 H1 — Тарифы (Rate Plans) + питание (Verpflegung)
**Зачем:** «тарифы» — ядро ТЗ (§7: 16 типов) и главный гэп. Сейчас у номера одна
цена + сезон/выходные. Отель продаёт один номер по нескольким тарифам: Basis,
Nicht-stornierbar (−10 %), mit Frühstück, Halbpension, Frühbucher, Flexibel.
**Что:** модель `RatePlan` (на тенанта, применима к юнитам): `name`, модификатор
цены (абсолютный ±€ или %), `meal_plan` (ohne/Frühstück/Halb-/Vollpension),
`cancellation` (flexibel/nicht-stornierbar + дата бесплатной отмены), флаг активности,
порядок. Гость на витрине выбирает тариф → снимок в `StayBooking.rate_*`. Цена тарифа
поверх посуточной (сезон/выходные сохраняются). Verpflegung как часть тарифа (а не
только Extra).
**Где:** `models.py` (+миграция), `pricing.quote_total_cents` (учёт модификатора),
`services.book_stay` (param rate_plan + снимок), `public_views`/`stay_detail.html`
(выбор тарифа, видимые условия отмены ДО оплаты — ТЗ §20), кабинет `units.py`
(CRUD тарифов), demo-kit.

### 🥈 H2 — Быстрый поиск с главной + список доступных номеров
**Зачем:** ТЗ §2.1 «форма быстрого поиска» на первом экране — главный UX-драйвер
прямого бронирования. Сейчас поиск только внутри карточки одного номера.
**Что:** секция «Suchformular» на главной (Anreise/Abreise/Erwachsene/Kinder/Zimmer)
→ страница результатов `/unterkunft/?von&bis&…` со ВСЕМИ доступными юнитами и ценой
«ab …€» за выбранный диапазон. Большая кнопка «Verfügbarkeit prüfen». Адаптив (ТЗ §12).
**Где:** новая секция в реестре `siteconfig.SECTIONS` + партиал; `unterkunft_index`
принимает GET-диапазон и считает котировки по всем юнитам; mobile-first.

### 🥉 H3 — Богатая карточка номера + условия отмены до оплаты
**Зачем:** ТЗ §2.2 — площадь, тип кроватей, удобства (Ausstattung), питание,
условия отмены, «похожие номера». Сейчас только name/type/описание/фото/цена.
**Что:** поля `StayUnit`: `area_sqm`, `bed_type`, `amenities` (чек-лист: WLAN/TV/
Balkon/Klimaanlage/Bad/…). Блок Ausstattung-иконок и условий отмены на странице
номера; «ähnliche Zimmer». Условия отмены/оплаты видимы до подтверждения (ТЗ §20).
**Где:** `models.py` (+миграция), `units.py` (форма), `stay_detail.html`, demo-kit.

### H4 — Промокоды и скидки (прямое бронирование) + loyalty-тариф
**Зачем:** ТЗ §4 «скидки; промокоды», §7.16 «Member/loyalty rate», §20 «скидка за
прямое бронирование». Есть `apps.loyalty` (Customer общий) — переиспользуем.
**Что:** промокод (фикс/процент, срок, лимит, min ночей) → применяется в брони stays,
снимок суммы скидки в `StayBooking`. Опц. «member rate» как тариф (H1) с условием.
**Где:** переиспользовать/расширить voucher-механику `apps.loyalty`; `book_stay` +
витрина (поле промокода), пересчёт total.

### H5 — Дети отдельно от взрослых
**Зачем:** ТЗ §2.1/§2.3 «взрослые + дети». Сейчас один `guests`.
**Что:** `StayBooking.adults`/`children`; вместимость = adults+children ≤ max_guests;
опц. детский тариф/бесплатно до N лет (просто, без возрастных сеток). Поиск (H2) и
карточка показывают раздельно.
**Где:** `models.py` (+миграция, дефолт adults=guests), формы/витрина.

### H6 — SEO отеля + Hausordnung
**Зачем:** ТЗ §13 микроразметка `Hotel`/`LodgingBusiness`, §2.10 правила проживания,
§14 юр-страницы (Impressum/Datenschutz/AGB уже есть на платформе).
**Что:** JSON-LD `LodgingBusiness`/`Hotel` (адрес/гео/цена «ab»/рейтинг из reviews) на
витрине; страница/секция Hausordnung (Check-in/out, отмена, депозит, дети, животные,
курение) — конструируемый блок. Open Graph/breadcrumbs — проверить, что уже есть.
**Где:** шаблоны витрины (head JSON-LD), новая секция/страница в siteconfig.

### H7 — Спецпредложения / пакеты (именованные)
**Зачем:** ТЗ §2.5/§8.3 — Frühbucher/Langzeit/Wochenende/Romantik/SPA-пакет как
витринные офферы (сейчас только сезон-цена). Частично закрывается H1 (тарифы) + H4
(промо). Отдельная подзадача — карточки «Angebote» на витрине + правила (напр.
Langzeit −X % от N ночей; Frühbucher при бронировании за N дней).
**Где:** небольшой слой над RatePlan/промо + секция «Angebote».

## 3. Рекомендуемый порядок реализации

MVP-ядро прямого бронирования малого отеля → затем доводка:

1. **H1 Тарифы + питание** (ядро ТЗ, наибольший гэп).
2. **H2 Поиск с главной + список номеров** (UX-конверсия).
3. **H3 Богатая карточка + условия отмены до оплаты** (доверие/право).
4. **H4 Промокоды/скидки** (прямое бронирование выгоднее OTA).
5. **H5 Дети** (мелко, но в ТЗ ядре поиска).
6. **H6 SEO + Hausordnung** (трафик/право).
7. **H7 Пакеты/офферы** (надстройка, опц.).

Каждая подзадача: отдельная ветка → push → CI зелёный → чекпоинт → строка в
`docs/build-log.md`. Demo-kit `hotel` обновляем по мере добавления фич, чтобы
«Pension Seeblick» демонстрировала новое из коробки.
