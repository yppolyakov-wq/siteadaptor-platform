# Hotel / Pension (A5 «Übernachtung») — план наполнения архетипа

Статус: 🟡 предложение (2026-06-22). Конвенция: docs до кода → разбивка владельцу →
реализация по подзадачам (ветка `claude/hotel-…` → CI зелёный → чекпоинт).

Архетип A5 (date-range размещение) уже на движке `apps.stays` (Track E, ~80 %).
Этот док — карта добивания до «полного отеля» по присланному ТЗ модуля
онлайн-бронирования, с приоритетом под **наш сегмент: маленький отель DACH**
(Pension, B&B, Ferienwohnung, kleines Hotel — 1–20 номеров).

**Установка владельца (2026-06-22):** план должен опираться на нашу разработку и быть
**в согласии с существующими модулями и агрегатором** — НЕ строим параллельную
инфраструктуру, а переиспользуем `apps.stays`, `core.Extra`, `apps.loyalty`,
`apps.finance`, `apps.notifications`, `apps.aggregator`. И: **для каждого архетипа —
свой агрегатор** (вертикальный портал). Для отеля это «Hotelsuche-Portal» — отдельная
подзадача H8 в этом плане; архитектура агрегатора её уже почти поддерживает
(см. §1.5), добиваем поиск по датам.

## 0. Принцип scope (что мы делаем и чего НЕ делаем)

ТЗ описывает спектр от «визитки маленькой Pension» до «большой сети с PMS». Наш
продукт — прямое бронирование для **малого** отеля. ТЗ §5.2 само относит к
«не обязательно для маленького отеля»: сложный Revenue Management, динамическое
ценообразование, сложная CRM/корп-тарифы, большой PMS. Берём это как ориентир.

**В scope (этот план):** богатые тарифы, питание, спецпредложения, промокоды,
быстрый поиск по датам с главной, дети, богатая карточка номера, условия отмены
видимые до оплаты, SEO-разметка отеля, Hausordnung, loyalty-тариф, рассылка гостям.

**Вне scope (заметка в roadmap, не делаем сейчас):**
- Метапоиск / Google Hotel Ads, полноценный Channel Manager (двусторонний),
  интеграция с большим PMS/замками/кассой — тяжёлая интеграционная инфраструктура;
  частично уже закрыто односторонним iCal (импорт занятости Booking/Airbnb + экспорт
  нашего фида). Полный CM — отдельный крупный модуль, не для MVP малого отеля.
- Роли сотрудников (Housekeeping/Revenue Manager/…), online-чекин, электронная
  регистрация гостей — для большой гостиницы (ТЗ §6). У нас уже есть `core.Membership`
  под роли — задел, но наполнение откладываем.
- Группы/корпоративные тарифы/агентские брони — редки у малого отеля; mini-вариант
  (заявка «для бизнеса») закрываем формой, без отдельного движка.

## 1. Что уже есть (фундамент, не переделываем)

- `StayUnit` (тип/quantity/цена·ночь/выходная цена/`SeasonRate`/min_nights/max_guests/
  deposit/manual-confirm/фото), `UnitBlock`, `ICalSource`, `StayBooking` (снимок цены,
  total, extras, deposit, payment_state, invoice_id).
- Анти-овербукинг по ночам (`services.book_stay`, `availability.range_available`).
- Витрина `/unterkunft/` (поиск по диапазону **в карточке номера**) → `/s/<code>/`.
- Extras (`core.Extra`, per-night) добавляются в брони на витрине и в total.
- Депозит онлайн (Stripe Connect), письма (created/confirmed/cancelled/reminder/owner),
  авто-Rechnung (`stay_to_invoice`, 7 % Beherbergung), iCal импорт/экспорт.
- Кабинет `/dashboard/stays/`: календарь загрузки, юниты, сезонные тарифы, блоки, iCal.
- Demo-kit `hotel` («Pension Seeblick»): 4 типа номеров, extras, season, deposit.

### Агрегатор (что уже умеет под отель)
- `AggregatorListing.KIND_STAY` — отдельный вид листинга для размещений; каждый
  активный `StayUnit` денормализуется в public-пул (`tasks.sync_stay_listing`:
  название/тизер/фото-или-логотип/цена·ночь/detail_url/гео), ресинк по сигналам
  save/delete и `reconcile_schema`. Гейт модулем `stays`.
- `AggregatorPortal.kind=vertical` + `business_type` — **вертикальный портал уже
  конфигурируется** (портал с `business_type="hotel"` сужает выдачу до отелей);
  резолвер по хосту (`middleware`) → `portal_views` с фасетами по городу/типу, гео-
  картой, избранным, отзывами/рейтингом (`BusinessReview`/`BusinessRating`).

## 1.5. Согласование с модулями и агрегатором (инвариант плана)

Каждая подзадача — НАДСТРОЙКА над существующим, без дублей:

| Потребность | Переиспользуем (не плодим) |
|---|---|
| Гость/идентичность | `promotions.Customer` (уже в `StayBooking.customer`) |
| Допуслуги | `core.Extra` (per-night) — тарифы H1 НЕ дублируют, а дополняют |
| Скидки/ваучеры/loyalty | `apps.loyalty` (voucher-механика, Customer общий) — H4 |
| Счета/НДС 7 % | `apps.finance` (`stay_to_invoice`) — без изменений |
| Письма/дедуп | `apps.notifications` + `stays.notifications` (idempotent) |
| Оплата/депозит | Stripe Connect (`billing.connect`, `stays.payments`) — H1/H4 поверх |
| Витрина/секции/SEO | `tenants.siteconfig` (реестр секций) — H2/H6 как секции |
| **Агрегатор** | `AggregatorListing.KIND_STAY` + `AggregatorPortal.kind=vertical` — H8 |

Правила: статусы — через FSM `.apply()`; внешние эффекты — Celery +
`idempotent_task`/`dedupe_key`; новые поля — миграции последовательные;
агрегатор/finance/billing — SHARED.

## 2. Гэпы по ТЗ → приоритизированные подзадачи

Приоритет = (леверидж для прямого бронирования малого отеля) × (близость к ТЗ-ядру
«даты/номера/тарифы/услуги/оплата/спецпредложения»).

### 🥇 H1 — Тарифы (Rate Plans) + питание (Verpflegung) — ✅ сделано (`stays/0008`)
**Зачем:** «тарифы» — ядро ТЗ (§7: 16 типов) и главный гэп. Сейчас у номера одна
цена + сезон/выходные. Отель продаёт один номер по нескольким тарифам: Basis,
Nicht-stornierbar (−10 %), mit Frühstück, Halbpension, Frühbucher, Flexibel.
**Что:** модель `RatePlan` (на тенанта, применима к юнитам): `name`, модификатор
цены (абсолютный ±€ или %), `meal_plan` (ohne/Frühstück/Halb-/Vollpension),
`cancellation` (flexibel/nicht-stornierbar + дата бесплатной отмены), флаг активности,
порядок. Гость на витрине выбирает тариф → снимок в `StayBooking.rate_*`. Цена тарифа
поверх посуточной (сезон/выходные сохраняются). Verpflegung как часть тарифа (а не
только Extra).
**Где:** `models.py` (+миграция), `pricing.quote_total_cents` (учёт модификатора),
`services.book_stay` (param rate_plan + снимок), `public_views`/`stay_detail.html`
(выбор тарифа, видимые условия отмены ДО оплаты — ТЗ §20), кабинет `units.py`
(CRUD тарифов), demo-kit.

### 🥈 H2 — Быстрый поиск с главной + список доступных номеров — ✅ сделано
**Зачем:** ТЗ §2.1 «форма быстрого поиска» на первом экране — главный UX-драйвер
прямого бронирования. Сейчас поиск только внутри карточки одного номера.
**Что:** секция «Suchformular» на главной (Anreise/Abreise/Erwachsene/Kinder/Zimmer)
→ страница результатов `/unterkunft/?von&bis&…` со ВСЕМИ доступными юнитами и ценой
«ab …€» за выбранный диапазон. Большая кнопка «Verfügbarkeit prüfen». Адаптив (ТЗ §12).
**Где:** новая секция в реестре `siteconfig.SECTIONS` + партиал; `unterkunft_index`
принимает GET-диапазон и считает котировки по всем юнитам; mobile-first.

### 🥉 H3 — Богатая карточка номера + условия отмены до оплаты — ✅ сделано (`stays/0009`)
**Зачем:** ТЗ §2.2 — площадь, тип кроватей, удобства (Ausstattung), питание,
условия отмены, «похожие номера». Сейчас только name/type/описание/фото/цена.
**Что:** поля `StayUnit`: `area_sqm`, `bed_type`, `amenities` (чек-лист: WLAN/TV/
Balkon/Klimaanlage/Bad/…). Блок Ausstattung-иконок и условий отмены на странице
номера; «ähnliche Zimmer». Условия отмены/оплаты видимы до подтверждения (ТЗ §20).
**Где:** `models.py` (+миграция), `units.py` (форма), `stay_detail.html`, demo-kit.

### H4 — Промокоды и скидки + самоотмена брони гостем — ✅ сделано (`stays/0011`)
**Зачем:** ТЗ §4 «скидки; промокоды», §7.16 «Member/loyalty rate», §20 «скидка за
прямое бронирование». Есть `apps.loyalty` (Customer общий) — переиспользуем.
**Что:**
- **H4a промокод** (фикс/процент, срок, лимит, min ночей) → применяется в брони stays,
  снимок суммы скидки в `StayBooking`. Опц. «member rate» как тариф (H1) с условием.
  Где: voucher-механика `apps.loyalty`; `book_stay` + витрина (поле промокода).
- **H4b самоотмена по ссылке** (✚ добавлено по решению 2026-06-22): подписанная ссылка
  «Stornieren» в письме, уважающая условия отмены тарифа (H1): для `flexible` —
  бесплатно до `free_cancel_days` (возврат депозита через Stripe Connect), для
  `non_refundable` — отмена без возврата. Делает условия отмены из H1 действенными.
  Где: паттерн подписанных токенов (как one-click-отписка), `StayBookingSM` cancel,
  `_refund_deposit`-ветка.

### H5 — Дети отдельно от взрослых (минимально) — ✅ сделано (`stays/0010`)
**Зачем:** ТЗ §2.1/§2.3 «взрослые + дети». Сейчас один `guests`.
**Scope (узкий, по решению 2026-06-22):** только `StayBooking.adults`/`children` для
вместимости (adults+children ≤ max_guests) и базы под Kurtaxe (H9). **БЕЗ возрастных
сеток и детских тарифов** — это Revenue-уровень, ТЗ §5.2 его не требует.
**Где:** `models.py` (+миграция, дефолт adults=guests), формы/витрина/поиск (H2).

### H9 — Kurtaxe / Tourismusabgabe (курортный сбор) — ✅ сделано (`stays/0010`)
**Зачем:** в большинстве городов DACH сбор за гостя за ночь обязателен по праву —
сейчас отсутствует. Важнее именованных пакетов (H7) и почти бесплатен.
**Что:** настройка на тенанта (сумма за взрослого за ночь, опц. дети бесплатно/льгота).
Отдельная строка в котировке (гости × ночи), в `total_cents` и в `stay_to_invoice`.
**Где:** настройка stays (Tenant/конфиг), `pricing` (отдельный компонент, не путать с
тарифом), снимок в `StayBooking`, витрина «inkl. X € Kurtaxe», Rechnung-строка.
**Зависит от H5** (нужны adults/children).

### H6 — SEO отеля + Hausordnung — ✅ сделано (`stays/0012`)
**Зачем:** ТЗ §13 микроразметка `Hotel`/`LodgingBusiness`, §2.10 правила проживания,
§14 юр-страницы (Impressum/Datenschutz/AGB уже есть на платформе).
**Что:** JSON-LD `LodgingBusiness`/`Hotel` (адрес/гео/цена «ab»/рейтинг из reviews) на
витрине; страница/секция Hausordnung (Check-in/out, отмена, депозит, дети, животные,
курение) — конструируемый блок. Open Graph/breadcrumbs — проверить, что уже есть.
**Где:** шаблоны витрины (head JSON-LD), новая секция/страница в siteconfig.

### H7 — Спецпредложения / пакеты (СВЁРНУТО в H1+H4)
**Решение 2026-06-22:** отдельную подзадачу не делаем — пакет = тариф (H1) с
включённым питанием + промо-правило (H4) (Frühbucher/Langzeit/Wochenende). Отдельный
витринный экран «Angebote» заводим, только если после H1–H4 чего-то реально не хватит.

### H8 — Агрегатор отелей (вертикальный портал «Hotelsuche»), 2 фазы
**Решение 2026-06-22 (де-риск):** разбить на H8a (дёшево, сразу) и H8b (тяжёлое):
- **H8a** — per-hotel карточки «ab €X» (агрегат cheapest unit) + сид/конфиг
  вертикального портала `hotels.<base>` поверх ГОТОВОГО `KIND_STAY` + `kind=vertical`.
  Без live-поиска по датам — статичная выдача + фасеты/гео/отзывы (всё уже есть).
- **H8b** — живой поиск по датам (Anreise/Abreise/Gäste) с обходом схем города и
  кешем (Redis, короткий TTL). Делаем, только убедившись, что H8a взлетел.
**Зачем:** установка владельца — «для каждого архетипа свой агрегатор». Для отеля это
портал-поиск по нескольким гостиницам города (мини-Booking для прямого бронирования
наших клиентов). База уже есть (см. §1.5): `KIND_STAY`-листинги + `kind=vertical`
портал. **Гэп — поиск по датам:** сейчас портал показывает листинги как статичные
карточки (цена·ночь снимком), без «введи Anreise/Abreise → доступные отели».
**Что (в согласии с агрегатором):**
1. **Per-hotel агрегация листинга.** Сейчас листинг = на каждый `StayUnit`. Для
   портала добавить агрегатный вид «отель» (cheapest unit → «ab €X»), чтобы карточка
   была про бизнес, а не про каждый номер. Решение: денорм поля в snapshot
   (`stay_min_price`, `stay_unit_count`) или отдельный агрегат при выдаче портала.
2. **Поиск по датам в портале.** Форма Anreise/Abreise/Gäste на портале
   `business_type=hotel`; на выдаче — отметка доступности/цены за диапазон.
   *Архитектурный нюанс:* доступность живёт в TENANT-схемах, листинги — в public.
   Подход: при поиске идём по схемам отелей города (как `reconcile_schema`) и
   считаем `availability.range_available`/`quote_total_cents`; кешируем результат
   (Redis, короткий TTL) — без тяжёлых кросс-схемных JOIN-ов. Малые объёмы города
   делают это дешёвым; при росте — материализованный календарь доступности.
3. **Hotel-фасеты:** тип (Pension/Hotel/Ferienwohnung), гости, удобства (из H3
   `amenities`), цена. Карта/«рядом» — уже есть (G8).
4. **Глубокая ссылка в прямое бронирование:** карточка отеля → `/unterkunft/?von&bis`
   уже с датами (стыкуется с H2). Прямое бронирование, без комиссии OTA.
**Где:** `aggregator/tasks.py` (`_stay_snapshot` + агрегат), `aggregator/models.py`
(denorm-поля/индекс), `aggregator/portal_views.py` (date-форма + доступность),
`aggregator/recommendations.py`/`context.py`, шаблоны портала. SHARED.
**Зависимости:** опирается на H2 (тот же date-quote-слой) и H3 (amenities-фасеты),
поэтому идёт после них.

## 3. Рекомендуемый порядок реализации

MVP-ядро прямого бронирования малого отеля → затем доводка → агрегатор поверх
(порядок уточнён 2026-06-22):

1. **H1 Тарифы + питание** ✅ (ядро ТЗ, наибольший гэп).
2. **H2 Поиск с главной + список номеров** (UX-конверсия; date-quote-слой нужен и H8b).
3. **H3 Богатая карточка + условия отмены до оплаты** (доверие/право; amenities → H8).
4. **H5 Дети (мини) + H9 Kurtaxe** (вместе — Kurtaxe зависит от adults/children).
5. **H4 Промокоды + самоотмена брони** (прямое бронирование выгоднее OTA; H4b делает
   условия отмены H1 действенными).
6. **H6 SEO + Hausordnung** (трафик/право).
7. **H8a Агрегатор отелей (карточки + портал)** → **H8b (поиск по датам)** —
   «свой агрегатор на архетип».
8. ~~H7 Пакеты~~ — свёрнуто в H1+H4.

Каждая подзадача: отдельная ветка → push → CI зелёный → чекпоинт → строка в
`docs/build-log.md`. Demo-kit `hotel` обновляем по мере добавления фич, чтобы
«Pension Seeblick» демонстрировала новое из коробки. Для H8 — сид demo-портала
`hotels.<base>` (kind=vertical, business_type=hotel), чтобы поиск работал из коробки.
