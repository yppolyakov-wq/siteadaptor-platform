# План: «настоящий анти-Битрикс» — кабинет/админка + онбординг

> Создан 2026-06-26 по фидбэку владельца. Цель (северная звезда): **чтобы ребёнок
> собрал и вёл магазин**. Конвенция «docs до кода». Source of truth этапа —
> `docs/archetype-ux-execution-plan.md`; этот док — отдельный трек (UX кабинета/онбординга),
> идём инкрементами AB1–AB5. Фундамент уже есть — переписываем подачу, а не модели.

## 0. Что болит (фидбэк владельца, 2026-06-26)
1. **Меню кабинета «как Битрикс»** — плоский технический список, не сгруппировано, не видно
   что важно; модули «под архетип» и «общие» свалены вместе.
2. **Нет нормального пошагового заполнения магазина при регистрации.** Мастер внутри есть
   (`/willkommen/`, 7 шагов), но «сырой и отсталый» и не связан с моментом регистрации.
3. **Непонятно, какой набор модулей доступен/нужен в этой модели (архетипе).**

## 1. Что УЖЕ есть (переиспользуем, не пишем заново)
- **Реестр модулей** `apps/core/modules.py::REGISTRY` (`ModuleSpec`): `key`, `label_de`,
  `icon`, `nav_items`, `recommended_for` (business_type → стартовый набор), `suited_for`
  (гибрид-подходит), `core`, `premium`, `description_de`, `storefront_*`. → метаданных для
  «модули по архетипу vs общие» уже достаточно.
- **Кабинет-нав** строится в `apps/core/context.py` (сайдбар из активных модулей + мобильный
  таб-бар `nav_primary`); шаблон `templates/tenant/_base_dashboard.html`.
- **Мастер онбординга** `apps/tenants/onboarding.py` (7 шагов, состояние в
  `site_config["onboarding"]`) + `apps/core/views.py::setup_view` + алиас `/willkommen/`.
- **Живое превью** витрины (`?preview=1` + `site_preview_draft`), демо-контент (`demo_kits`/
  `load_demo`), on-canvas редактор (Спринт E).

## 2. Принципы анти-Битрикс (держать во всех инкрементах)
- **Язык задач, не сущностей:** «Bestellungen annehmen», не «Modul orders».
- **Прогрессивное раскрытие:** не вываливать 15 модулей сразу; показывать активное + «➕ ещё».
- **Дефолты вместо пустых полей:** везде демо-примеры, пользователь правит, а не создаёт с нуля.
- **Живое превью на каждом шаге.** Обратимость (Undo/сброс демо). Нельзя «сломать».
- **Мобайл-first, один экран = одно решение.** Крупные кнопки, всегда «Назад»/«Пропустить».

## 3. Направления и инкременты

### AB1 — Пересборка меню кабинета (группировка по задачам) [M, без моделей]
**Проблема:** плоский список модулей в сайдбаре, технический язык.
**Решение:** ввести **группы навигации** (в реестре/контексте), сайдбар рисует заголовки групп:
- **«Mein Geschäft»** — Übersicht, Сайт/Витрина (билдер), Vorschau.
- **«Verkaufen»** (по архетипу, только активные) — Katalog, Bestellungen, Termine, Stays,
  Events, Aufträge/Angebote.
- **«Kunden & Marketing»** — Kunden (CRM), Aktionen, Treue, Newsletter, Bewertungen.
- **«Einstellungen»** — Einstellungen, Domains, Module, Abrechnung.
- Подписи пунктов — **в языке задач** (новое поле `ModuleSpec.task_label` или маппинг).
- В сайдбаре только активные модули + заметная **«➕ Funktion hinzufügen»** → страница «Module».
- Мобильный таб-бар: 4–5 ключевых + «Mehr».
**Файлы:** `modules.py` (группа на модуль), `context.py` (сгруппированный nav),
`_base_dashboard.html` (заголовки групп). Тест: nav-группы в контексте + рендер заголовков.

### AB2 — Страница «Module» v2 (модули по архетипу vs общие) [S–M]
**Решение:** разбить страницу выбора функций на секции:
- **«Für Ihr Geschäft empfohlen»** — `recommended_for ∪ suited_for` по `business_type`
  (преднажато для пресета).
- **«Weitere Funktionen»** — универсальные (`recommended_for`/`suited_for` пусты) + прочие.
- **«Premium»** — `premium=True` (бейдж тарифа).
- Карточка модуля: иконка, **что даёт** (`description_de`), **«Gut für: Hotel, Pension»**
  (из recommended/suited), тумблер вкл/выкл, бейдж Core/Premium (Core — без тумблера).
**Файлы:** view страницы «Module» (в `apps/core/views.py`), её шаблон. Тест: секции/бейджи/
гейтинг по business_type.

### AB3 — Мастер онбординга v2 («ребёнок справится») [M] 🚧 (шаг 1 визуализирован ✅ df4bba8)
**Проблема:** мастер сырой, без превью/демо-дефолтов, технический.
**Решение (поверх существующих 7 шагов):**
- **Прогресс-бар** «Schritt N von M» + крупные кнопки, всегда «Zurück»/«Überspringen», автосейв.
- **Демо по умолчанию на каждом шаге:** уже стоят примеры (3 товара/услуги с фото-заглушками,
  баннер, часы) — правишь, а не создаёшь с нуля (переиспользовать `demo_kits`/`load_demo`).
- **Живое превью** витрины на каждом шаге (iframe `?preview=1`, как в билдере).
- Шаги — в языке задач: 1) Was verkaufst du? → 2) Wie sieht es aus? (тема/цвет) → 3) Was zeigen
  wir? (модули, преднабор архетипа) → 4) Deine ersten 3 Angebote (с примерами) → 5) Banner & Logo
  → 6) Kontakt/Öffnungszeiten/Recht (Impressum авто) → 7) Fertig — «вот ссылка, поделись».
**Файлы:** `setup_view` + шаги-шаблоны, `onboarding.py`. Тесты: прогресс, демо-дефолты, превью-линк.

### AB4 — Чек-лист готовности сайта на дашборде [S–M]
**Решение:** после мастера на `/dashboard/` — карточка **«Dein Site ist zu X% fertig»** с
пунктами-ссылками: «Foto hinzufügen», «Öffnungszeiten», «erste Aktion», «Impressum prüfen».
% и пункты считаются из заполненности `site_config`/каталога (нет фото → пункт активен).
Мотивация + прямой путь к действию (анти-«пустой кабинет»).
**Файлы:** dashboard-вью + партиал чек-листа + хелпер «completeness(tenant)». Тест: расчёт пунктов.

### AB5 — Связать регистрацию → мастер [M, HIGH-RISK, гейт владельца] ✅
**Проблема:** signup/provisioning и `/willkommen/` разорваны → после регистрации пустой кабинет.
**Сделано:** redirect-gate в самой вьюхе `dashboard` (без правок signup/provisioning): владелец с
нетронутым мастером (шаг 1, без пропусков, не завершён) при заходе на `/dashboard/` редиректится
в `setup`. Любое действие в мастере (Weiter/Überspringen/Zurück) или его завершение снимает гейт;
остальная навигация кабинета не гейтится. Тесты: свежий → редирект; тронутый/skipped/completed →
рендер. Минимальный и обратимый — без миграций и без изменения создания тенанта.

## 4. Порядок и риски
Порядок: **AB1 → AB2 → AB3 → AB4 → AB5**. AB1–AB4 — UI/контекст/хелперы, низкий риск, без
моделей/миграций (кроме, возможно, маленького поля `task_label` — можно без миграции, в реестре).
AB5 — high-risk (signup/provisioning), последним, аккуратно, гейт владельца.
Каждый инкремент: ветка → ruff (раздельно) → build:css → pytest → push → CI зелёный → FF-мерж →
отметка в `archetype-ux-execution-plan.md` + строка в `build-log.md`.

## 5. Definition of done трека
Кабинет читается «с порога» (группы/язык задач), модули видно «что для моего бизнеса», новый
владелец доходит до живого сайта за один линейный мастер с демо-дефолтами и превью, на дашборде
есть понятный чек-лист «что доделать». Метрика-ориентир: «ребёнок 10 лет соберёт витрину без
подсказок».
