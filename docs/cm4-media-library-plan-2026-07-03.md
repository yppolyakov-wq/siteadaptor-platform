# CM-4 — медиа-библиотека тенанта (план до кода, 2026-07-03)

Четвёртый инкремент контент-хаба. Разведка агентом (полная карта — в отчёте),
факты сверены с кодом.

## 1. Ключевые факты разведки

- **Единой точки входа загрузок ДВЕ**: `catalog.images.save_product_image`
  (все FileRef-потоки: 12+ мест — товары/акции/события/номера/услуги/блог/посты/
  Extra/Resource/hero/галереи site_config) и отдельный `jobs.JobPhoto.ImageField`
  (вне механизма — не трогаем в CM-4).
- **Реестра медиа НЕТ**: фото рассыпаны по JSONField'ам; посчитать/показать все
  файлы тенанта нельзя. FileRef: `{id,url,path,alt{de,en},mime_type,size,
  is_primary,sort_order}`; демо-FileRef без `path` (SVG-вьюха, не storage).
- **alt** есть в схеме, но пишется пустым, редактора нет; рендерится только в
  `sections/_gallery.html`; в `_media_gallery` alt = имя сущности, миниатюры "".
- **Orphans**: файл удаляется только явными per-image операциями; при удалении
  сущности (везде soft-delete) и перезаписи одиночных URL-полей (hero/logo/
  Resource.photo/Extra.image) старые файлы сиротеют. Дедупа нет (uuid-имена).

## 2. Дизайн: реестр ПОВЕРХ, FileRef-копии остаются как есть

Полная перелинковка потребителей на центральную модель — L и горячо. Вместо
этого: **`MediaAsset` — индекс** (TENANT-модель: `path` unique, url, folder,
mime_type, size, alt {локаль: str}, created), наполняемый в единственной точке
входа: `save_product_image` после сохранения создаёт запись,
`delete_stored_image` — удаляет. Существующее — backfill-командой (обход карты
мест из разведки §2: 4 галереи + 5 одиночных FileRef + site_config-галереи;
demo-FileRef без path пропускаются). Копии FileRef в сущностях НЕ трогаем —
реестр это индекс, не источник рендера.

**alt**: источник правды остаётся в FileRef (рендер не меняется). Редактор alt
на странице «Medien» пишет в РЕЕСТР и write-back'ом обновляет найденные
FileRef-копии по `path` (тот же обход, редкая операция). Заодно `_media_gallery`
миниатюры получают `alt` из FileRef (`img.alt.de`) — закрывает a11y-гэп.

## 3. Слайсы

- **A. Модель+хук+backfill:** `MediaAsset` (миграция; app — НОВОЕ TENANT-app
  `apps.media` НЕ создаём — кладём в `apps/core/models.py`? НЕТ: медиа — контент;
  решение: в `apps.catalog` рядом с images.py тоже плохо. Выбор: **apps/core**
  (инфраструктурная модель, как Extra) — без нового приложения и TENANT_APPS-
  правок). Хук в `save_product_image`/`delete_stored_image` (fail-safe: ошибка
  реестра не роняет загрузку). Management-команда `backfill_media_registry`
  (per-schema; идемпотентна по unique path).
- **B. Кабинет «Medien»** (`/dashboard/medien/`): сетка ассетов (фильтр по
  folder, превью, размер), редактор alt (реестр + write-back по path), удаление
  файла ТОЛЬКО если обход не находит использований (защита от битых ссылок) —
  кнопка «Löschen (ungenutzt)». NavItem — в core-набор кабинета (страница
  доступна всем — медиа есть у любого архетипа; без нового модуля).
- **C. (следующие слайсы, НЕ сейчас):** медиа-пикер «из библиотеки» в билдере
  (hero/галереи/C-блоки), пер-слайд alt-инпут на канве, дедуп по контент-хэшу,
  чистка orphan'ов при перезаписи одиночных полей.

## 4. Замки/тесты

save_product_image создаёт запись (и не падает при ошибке реестра); backfill
идемпотентен и покрывает галерею+одиночное+site_config; удаление занятого
файла отклоняется, незанятого — удаляет storage+реестр; alt write-back меняет
FileRef-копию; существующие тесты фото-эндпоинтов (реестр прозрачен для них).

## 5. Риски

- Хук в save_product_image — общая точка ВСЕХ загрузок: строго fail-safe
  (try/except, реестр вторичен к файлу).
- Write-back alt обходит JSONField'ы — карта мест закреплена в одном модуле
  (`apps/core/media_registry.py`), при добавлении новых мест хранения — дописать
  туда (заметка в CLAUDE.md §5 не нужна — карта используется и backfill'ом,
  рассинхрон всплывёт тестом).
