# Архетип «Ретрит» — план наполнения (A6+)

> Статус: **ЧЕРНОВИК на согласование** (2026-06-24). Конвенция: docs до кода.
> Source of truth по вертикалям — `micro-business-verticals.md`; по сделанному —
> `build-log.md`. Этот документ — оценка ТЗ владельца + приоритизированный бэклог
> подзадач для архетипа «Ретрит». Кодим только после подтверждения порядка.

## 0. Контекст и позиционирование

Платформа — **мультитенантный SaaS для микробизнеса DACH** (один тенант = один
сайт на субдомене, ~39 €/мес, витрина без аккаунта для конечного клиента, без
трекинг-куки, Double-Opt-In по UWG §7). Это важная рамка: ТЗ владельца описывает
местами **маркетплейс-платформу уровня BookRetreats/OUM** (роли Участник →
Организатор → Преподаватель → Администратор → Партнёр по размещению, десятки
локаций, мультивалюта, подбор авиабилетов). Часть этого **противоречит** модели
«сайт одного бизнеса» и относится к Stage 3 (маркетплейс) либо к агрегатору, а не
к архетипу. Ниже честно делю ТЗ на: *уже есть* / *вписывается, делаем* / *вне
рамок архетипа*.

Ретрит технически = **A5 (размещение по датам, `apps.stays`) ⊕ A6 (событие с
билетом/программой/ростером, `apps.events`)**. Сегодня кит `retreat` стоит на A6
(events + 1:1-booking + shop). Основная стратегическая развилка (см. §4, R-mix) —
оставаться на «событии с тирами» или подключить полноценный date-range stays для
многодневного проживания с выбором типа комнаты.

## 1. Что уже есть (база A6, готово в коде)

Движок `apps.events` + кит `retreat` покрывают значимую часть ТЗ:

- **Событие на дату** с вместимостью и анти-овerselл мест (атомарно), `capacity=0`
  = безлимит (фестиваль). `seats_left` / `is_sold_out`.
- **Ценовые тиры билета** (`Event.tiers` → Frühbucher / Standard / Mehrbett) —
  частично закрывает «варианты размещения» (§5/§7 ТЗ) на уровне цены.
- **Богатый ретрит-лендинг** (`Event.details`, `apps/events/details.py`): promise,
  idea, for_whom, includes, **venue**, **accommodation** (список), **food**,
  **hosts** (имя/роль/фото), price_includes/excludes, price_note, **bring**, faq,
  testimonials. Рендер — `templates/storefront/event_detail.html` (2 колонки,
  программа, тиры, ведущие, отзывы, FAQ, «осталось N мест»).
- **Программа** (`Event.program`) — агенда по дням/часам.
- **Анкета участника** (`Event.questions` → `Ticket.answers`) — произвольные поля
  (аллергии, опыт и т.п.) задаёт организатор.
- **Extras к билету** (`core.Extra`, scope events): Bio-Mittagessen, Einzelzimmer-
  Zuschlag, Yogamatte — частично закрывает питание/доплаты (§13).
- **Онлайн-оплата** билета через Stripe Connect (на счёт бизнеса), `payment_state`.
- **Уведомления** Email + **Telegram** клиенту (`apps/telegram`).
- **Ростер участников** + экспорт CSV (кабинет организатора), CRUD событий,
  ручное подтверждение/отмена билетов.
- **ЛК клиента** (`customer_account`) включён в ките — бронирования/счета.
- **Галерея**, **отзывы на витрине** (`reviews_seed`), **FAQ**, Trust, команда.
- **Агрегатор**: события уже листятся (`aggregator/...stay_event_listings`).
- **Site Builder (M20)**: главная с hero «ближайшее событие», секции,
  адаптив/мобильный таб-бар — закрывает §3/§22 ТЗ почти целиком конструктором.
- **Подарочные сертификаты** (`apps/loyalty/gift.py`) — есть в платформе,
  переиспользуемы (сейчас не подключены к events).
- **1:1-услуги** (booking) + **shop**-каталог (мерч для практики).

**Вывод:** MVP из §23 ТЗ (главная, страница ретрита, бронь, оплата, галерея,
отзывы, контакты, Telegram) — **по сути уже собирается** из готовых кирпичей.

## 3. Вне рамок архетипа (НЕ делаем сейчас — обоснование)

- **Мультивалюта EUR/USD/GBP/INR (§9).** Биллинг, Stripe Connect, finance/DATEV,
  PAngV — всё в EUR (DACH-позиционирование). Мультивалюта — сквозная переделка
  ядра, не фича архетипа. → отклонить (или Stage 3).
- **Платформа с ролями Участник→Организатор→Преподаватель→Админ→Партнёр,
  маркетплейс десятков локаций (финал ТЗ).** Это **Stage 3 (маркетплейс-корзина
  M14)** + агрегатор, а не архетип «сайт одного ретрит-центра». Архитектурные швы
  (`core.Membership`/роли) уже заложены; полноценный маркетплейс — отдельная стадия.
- **Подбор авиабилетов (§доп, §20 Wise).** Travel-агрегация/affiliate — чужой
  домен, не микробизнес-сайт. → отклонить.
- **Поиск попутчика / выбор места в комнате (§доп).** Очень нишево, высокая
  сложность UX, низкая массовость для DACH-микробизнеса. → отложить (заметка в roadmap).
- **Zoom / Google Calendar / Mailchimp / PayPal / Wise интеграции (§20).** Точечно:
  рассылки уже есть свои (NewsletterCampaign, UWG-совместимо) → Mailchimp не нужен;
  PayPal — дублирует Stripe; Zoom/онлайн-ретриты — отдельный мини-трек (ссылка-доступ
  после оплаты), низкий приоритет. iCal-экспорт события (вместо Google Cal API) —
  дёшево, можно добавить к P6.
- **Авто-создание Telegram-группы после оплаты (§доп).** Интересно, но требует
  bot-admin прав и модерации; отложить.

## 4. Рекомендованный порядок (по ценности/стоимости)

Принцип: сперва дёшево закрыть видимые дыры воронки продаж, затем крупные модули.

1. **R1 — Waitlist + структурированная регистрация (P1+P3).** Размер **M**.
   Прямо влияет на конверсию sold-out и качество данных участника. Низкий риск,
   переиспользует паттерн promotions waitlist.
2. **R2 — Каталог: типы/направления + фильтры (P2).** Размер **M**. Делает
   витрину «каталогом ретритов» из ТЗ §4; основа для SEO-страниц (§21) и
   подбора по интересам.
3. **R3 — Преподаватели как сущность (P5) + календарь/iCal (P6).** Размер **M–L**.
   «Учителя» и «Календарь на год» — заметные блоки ТЗ §3.
4. **R4 — Частичная оплата/депозит билета (P4) + подарочный сертификат (P11).**
   Размер **M**. Снимает барьер дорогого ретрита (платёж частями).
5. **R5 — Многодневное проживание ↔ stays с типом комнаты (P8).** Размер **L**.
   Самый крупный; делать, когда ядро событий укреплено. Решает «реальный ретрит
   с ночёвкой и выбором размещения».
6. **R6 — Карта на витрине (P7) + памятка-PDF (P9) + корп-запрос (P13).** Размер
   **S–M** каждый. Догоняющие штрихи.
7. **R7 — Блог (P10).** Размер **M**. Общеплатформенный, не только ретрит;
   делать, если контент-SEO в приоритете.
8. **Отложено:** WhatsApp (P12, нужен аккаунт), онлайн-ретриты/Zoom, попутчик,
   авто-TG-группа — заметки в `roadmap-next-sprints.md`.

**Демо-кит `retreat`** обогащаем по мере фич (по нескольку примеров на фичу, как в
hotel-ките): waitlist-событие, типы/преподаватели, событие с проживанием.

## 5. Открытые вопросы к владельцу (перед кодом)

1. **Стартовая подзадача:** начинать с **R1** (waitlist+регистрация — дёшево и
   конверсионно, рекомендую) или сразу с крупного **R5** (проживание+комнаты)?
2. **R5 (комнаты):** строить на связке `events ⊕ stays` (мощно, но сложнее) или
   достаточно расширить тиры до «тип комнаты + наличие на событие» (легче, без stays)?
3. **Объём архетипа vs Stage 3:** подтверждаем, что роли/маркетплейс/мультивалюта —
   **вне** этого этапа (только швы), а делаем «богатый сайт одного ретрит-центра»?
4. **Блог (R7):** нужен в этом этапе или отдельно (он общеплатформенный)?
