## A. «Где это?» — уже есть, плохо находимо (ответы + доработка находимости)

### FB-1 · Food labeling в карточке товара — ✅EXISTS (S)
- **Где:** форма товара → вкладка **«Kennzeichnung» (Labeling)** → блок `data-food-section`
  (`templates/catalog/product_form.html:57,91-93`).
- **Почему не видно:** гейт `show_food_labeling = business_type ∈ FOOD_BUSINESS_TYPES`
  = только **{bakery, butcher, grocery, restaurant, cafe}** (`apps/catalog/views.py:37,46`).
  Для retail/online_shop/clothing/hotel/friseur/handwerker/… секция скрыта. ПЛЮС вся
  вкладка «Kennzeichnung» скрыта в **Простом режиме** (`ui_simple`).
- **Задача:** (а) короткое пояснение «доступно для еды/гастро» ИЛИ (б) кнопка «Пищевую
  маркировку показать» для любого типа (тумблер в site_config), чтобы не-еда тоже могла
  включить. Решить с владельцем: авто-по-архетипу vs ручной тумблер.

### FB-2 · Customize columns в Kanban-доске — ✅EXISTS (S)

## B. Реальные пробелы — нет UI

### FB-3 · Условия/запрет перемещения карточек в Kanban — ⚠️GAP (L)
- **Сейчас:** правила перехода = FSM `transitions` в КОДЕ (`apps/orders/state_machine.py:12`
  и аналоги booking/stays/jobs/events). Панель «Customize columns» ЯВНО пишет «rules for
  moving cards stay the same» — двигать можно только по разрешённым FSM-переходам, снап-бэк
  на 409.
- **Почему сложно:** переходы завязаны на побочки (письма клиенту, возврат склада,
  запись выручки, un-redeem промокода — `on_transition`). Произвольные правила их ломают.
- **Задача (обсудить объём):** НЕ трогая FSM-инварианты — дать владельцу конфиг
  «какие ручные переходы разрешены на доске» (подмножество имеющихся) + опц. запрет
  отдельных. Полноценный конструктор статусов/правил — большой отдельный трек.

### FB-4 · Настройка статусов заказа — ⚠️GAP (M переименование / L новые статусы)
- **Сейчас:** статусы = состояния FSM (`new/confirmed/ready/picked_up/shipped/cancelled/
  returned`), хардкод. UI нет.
- **Задача:** (а) переименование ЛЕЙБЛОВ статусов per-тенант — M (как board labels, но на
  статусы; рендерить кастомный лейбл в списках/письмах). (б) Добавление НОВЫХ статусов — L
  (завязано на письма/финансы/переходы, требует движка). Начать с (а).

### FB-8 · Единый «отдел продажных сущностей» (номера/услуги/товары в одном месте) — ⚠️GAP (L)
- **Запрос:** «непонятно где настройка товара для гостиницы — это сущность товара/
  ассортимент как другие услуги; они все должны быть в одном отделе продажных сущностей».
- **Сейчас:** управление РАЗНЕСЕНО по приложениям: товары `/dashboard/catalog/`, услуги
  `/dashboard/booking/`, номера `/dashboard/stays/`, события `/dashboard/events/`.
  Витрина уже унифицирована (мастер-трек `unified-sellable-entity`: единая деталь/листинг/
  карточка через контракт `SellableEntity`), но КАБИНЕТ-управление ещё по-старому силосно.
- **Задача (крупная):** единый раздел «Sortiment / Angebote» — общий листинг всех
  продаваемых сущностей тенанта (товар/услуга/номер/событие) с фильтром по типу и
  переходом в нужную форму; ИЛИ единая точка входа «＋ Neu» → выбор типа. Опирается на
  существующий реестр `SellableEntity` (`apps/core/sellable.py`). Source of truth —
  `docs/unified-sellable-entity-master-track-2026-06-30.md` (кабинет-часть там не закрыта).

## C. Баги

### FB-13 · Иконка «сменить фото» пропадает при наведении — 🐞BUG (S)
- **Где:** on-canvas фото-контролы `📷/🗑/＋` — `templates/storefront/_media_gallery.html:25-33`
  (класс `sf-photo-edit`, эмодзи + `hover:text-indigo-600`).
- **Гипотеза:** при hover меняется `text-*` цвет (эмодзи его игнорируют, но контейнер/
  родительский `hover:opacity-90` у плитки-превью, строка :20, может гасить оверлей) —
  значок «исчезает». Нужна репро в Chromium + правка CSS (убрать конфликт opacity/цвета,
  дать иконке свой фон/opacity независимо от hover плитки).
- **Задача:** воспроизвести, поправить CSS, headless-проверка.

### FB-14 · Внешняя ссылка не открывает внешний сайт («replicate.com refused to connect») — 🐞BUG? NEEDS-REPRO (S)
- **Проверено:** в коде (templates/apps) `replicate.com` НЕ упоминается (только в
  `SOURCES.md`/доках, не user-facing). Значит это либо (а) ссылка открывается в iframe/
  превью, а внешний сайт запрещает встраивание (X-Frame-Options → «refused to connect»),
  либо (б) владелец сам открывал replicate.com (отзыв токена) и это его сеть.
- **Нужно от владельца:** где именно кнопка/ссылка (какая страница кабинета/витрины,
  что за ссылка). Тогда чиним `target="_top"`/`rel` или открытие в новой вкладке. Кандидаты
  на iframe-контекст — превью сайта (`site_preview.html`, «View demo site»).

  (`product_form.html:81`, `promotion_form.html:115`, `booking/services.html:33,85`).
  Превью существующих — с ★/✕, но «＋»-тайла для добавления НЕТ (на канве ＋ есть —
  `_media_gallery.html:33`, но на кабинет-форме нет).
- **Задача:** единый визуальный «＋ Foto»-тайл В НАЧАЛЕ ряда превью (галерейный паттерн),
  консистентно на товаре/акции/услуге/номере/событии. Партиал `tenant/_photo_add.html`
  поверх существующих upload-эндпоинтов. Инвариант W0 (поля в DOM).

### FB-6 · Фото категорий и субкатегорий — 🆕FEATURE (M, ⚠️миграция)
- **Сейчас:** у `Category` (models.py:25-58) поля images **НЕТ** (в отличие от `Product.images`,
  строка 75 — это Product с 59). Форма категории (`category_form.html`) фото-UI не имеет
  (0 упоминаний). Витрина плитки категорий фото не рендерит.
- **Задача:** (1) миграция — `Category.images` JSONField (FileRef-конверт как у Product,
  аддитив); (2) форма категории — upload + «＋»-тайл (FB-5) + ★/✕; (3) витрина — фото на
  плитке категории/подкатегории (там где рендерятся категории); (4) субкатегории — та же
  механика (Category.parent). Загрузка/ресайз — переиспользовать `catalog/images.py`.
