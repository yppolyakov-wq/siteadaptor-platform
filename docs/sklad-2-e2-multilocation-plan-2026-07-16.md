# Склад-2 E2 — Мультисклад (Mehrlager): план-разбивка

> **Статус: DRAFT (2026-07-16).** Последний эпик волны U-D2W (владелец: «все 3»,
> порядок E1→E3→E2). E1 (Chargen/MHD+FEFO) и E3 (Закупки/M12 v1) закрыты. Самый
> инвазивный эпик — «ломает модель одного счётчика» (план §2), поэтому режем так,
> чтобы счётчик НЕ сломать: та же архитектура Вариант A, что спасла E1.

## 1. Ключевое решение: локация = разбивка ПОВЕРХ счётчика (Вариант A, как партии)

**НЕ переписываем движки.** `stock_quantity` остаётся ИТОГО-истиной (все продажи/
брони/anti-oversell работают как сегодня, безо всяких правок атомиков). Локации —
измерение РЯДОМ, выведенное из леджера:

- `StockMovement += location` (FK, nullable; NULL = «основной склад» — вся история
  до E2 автоматически валидна).
- Баланс per-location = Σ дельт леджера по локации (как `ledger_balance` сегодня).
- Σ по всем локациям ↔ счётчик — та же реконсиляция, что леджер↔счётчик и Σlot↔счётчик.
- **Umlagerung (переброс)** = пара движений в одной atomic: `transfer_out(−N, loc A)` +
  `transfer_in(+N, loc B)` — счётчик НЕ двигается (Σ=0), только разбивка.
- Продажа/заказ склад-локацию НЕ выбирает (v1): расход пишется с location=NULL →
  «основной». Продажа с локации — v2 (потребует выбора в checkout — отдельное решение).

Плюс: полная совместимость, ноль правок в orders/jobs/booking. Минус: в v1 продажи
списываются «с основного» — для микробизнеса с одной точкой продаж это и есть реальность
(вторая локация = склад/производство/фургон).

## 2. Модели (⚠️ миграция `inventory/0004`, аддитив)

- **`StockLocation`**: `name`, `is_default` (ровно одна; NULL-леджер читается как
  default), `note`, `is_active`, Timestamped. Создаётся лениво: пока локаций 0 —
  весь мультисклад-UI скрыт, поведение сегодняшнее.
- **`StockMovement.location`** FK (SET_NULL, nullable, blank) + kinds `transfer_in`/
  `transfer_out`.
- (НЕ добавляем `Lot.location` в v1 — партии остаются глобальными; связка партий с
  локацией — v2 по спросу.)

## 3. Сервисы (`inventory/services.py` / новый `locations.py`)

- `location_balance(entity, location)` — Σ дельт леджера по локации (NULL = default).
- `location_rows(entity)` — разбивка по локациям + «не распределено» (счётчик − Σ).
- `transfer(*, product, variant, src, dst, qty, actor)` — пара движений atomic,
  клампы по балансу src (не даём увести в минус).
- `receive_*` / `apply_manual_movement` += опц. `location` (приёмка на локацию).

## 4. Кабинет

- Тумблер не нужен: сам факт «локаций > 1» включает UI (ленивая активация).
- `/dashboard/stock/`: секция «Standorte» (создать локацию; список), у сущности —
  разбивка по локациям (drill-down в существующей таблице), форма Umlagerung
  (сущность + из → в + Menge).
- Приёмка (stock + purchasing): селектор локации (виден при локациях > 1).

## 5. Фазы

- **E2.1 (⚠️ миграция `inventory/0004`)** — модели + сервисы + transfer + замки
  (баланс per-location, переброс не двигает счётчик, реконсиляция Σloc↔счётчик).
- **E2.2** — кабинет: Standorte-CRUD + Umlagerung + разбивка (UI виден только при
  локациях > 1) + селектор локации на приёмках (stock + purchasing).
- **E2.3 (опц.)** — демо (вторая локация «Lager hinten» у retail-кита) + отчёт.

## 6. Риски

- Инвариант: счётчик = итого, НИКОГДА не пишем его из локационной логики (только
  существующие пути). Переброс Σ=0 — единственная новая запись леджера.
- Валидность старой истории: NULL location = default — без бэкфилла.
- Не захламлять: 0 локаций → UI не показываем вообще (большинству тенантов
  мультисклад не нужен).

## 7. Вне v1

Продажа с выбором локации (checkout), `Lot.location`, per-location Meldebestand,
инвентаризация per-location, отчёты по локациям.
