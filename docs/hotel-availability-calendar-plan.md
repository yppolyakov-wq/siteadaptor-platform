# План: визуальный календарь наличия (A5 отель) — самый сильный рычаг конверсии

> Создан 2026-06-26. Конвенция «docs до кода»: крупная доработка (M–L) — отдельным
> план-доком до кода. Source of truth этапа — `docs/archetype-ux-execution-plan.md`
> (§ F-A5). Идём строго инкрементами; один инкремент = ветка/коммит → CI зелёный → мерж.

## 1. Зачем
Сейчас гость на странице номера (`/unterkunft/<pk>/`) выбирает даты «вслепую» через
два инпута и узнаёт о занятости только после сабмита (`_quote → unavailable`). Все
сильные конкуренты (Booking, HRS, FeWo-direkt) показывают **визуальный календарь**:
свободные/занятые ночи видны сразу, выбор диапазона — кликом. Это снимает трение и
прямо повышает конверсию брони. Рынок-обоснование — `docs/market-analysis/*` (A5).

## 2. Что уже есть (переиспользуем, без новых моделей)
- `apps/stays/availability.py::range_available(unit, von, bis, needed)` — пер-ночная
  проверка занятости (активные `StayBooking` + `UnitBlock`, с учётом `quantity`/`rooms`).
- `apps/stays/pricing.py::nightly_price_cents(unit, day)` — цена за ночь (для тултипа/
  «ab …€» в ячейке, опц.).
- Витрина номера `unterkunft_unit` уже считает `_quote`, тарифы, Kurtaxe; форма дат —
  GET (`von`/`bis`). Календарь встроится рядом с формой выбора дат.
- `StayUnit.quantity` (мультиномерность, G5) → ячейка показывает «N frei», а не просто
  занято/свободно.

## 3. Принцип реализации
**Server-rendered месяц-сетка + HTMX-навигация по месяцам, без SPA, без новых моделей,
без трекинг-куки.** Календарь — это представление поверх существующей occupancy-логики.

### 3.1 Данные (новый helper в `availability.py`)
```python
def month_availability(unit, year, month, *, today=None) -> list[dict]:
    """Список дней месяца: {date, in_past, free (quantity − occupancy), is_free}.
    occupancy — та же логика, что range_available, посчитанная разом на месяц
    (один проход по StayBooking+UnitBlock, без N запросов)."""
```
- Один запрос броней/блоков на месяц (диапазон 1-е…последнее число) → словарь
  `date → occupied`, затем `free = quantity − occupied`, `is_free = free > 0`.
- Прошлые дни (`date < today`) помечаем `in_past=True` (некликабельны).

### 3.2 Вьюха (HTMX-партиал для перелистывания)
- Новый GET-эндпоинт `unterkunft_unit_calendar(request, pk)` → рендер партиала
  `storefront/_stay_calendar.html` с `?year=&month=` (по умолчанию текущий месяц).
- Ответ — только партиал (для `hx-get` swap при «‹/›» переключении месяца).
- Гейт — модуль `stays` активен (как `unterkunft_unit`).
- Маршрут: `path("unterkunft/<uuid:pk>/kalender/", ...)` в `config/urls_tenant.py`.

### 3.3 Шаблон `storefront/_stay_calendar.html`
- Сетка 7×N (Mo–So, немецкая неделя), заголовок «Monat Jahr» + кнопки ‹ › (hx-get
  на тот же эндпоинт с соседним месяцем, `hx-target` = контейнер календаря).
- Ячейка-день:
  - прошлое/вне сезона → серый, неактивный;
  - свободно → кликабельно (зелёный акцент), `data-date="YYYY-MM-DD"`, при `quantity>1`
    подпись «N frei»;
  - занято (free=0) → перечёркнуто/серое, некликабельно.
- Выбор диапазона — лёгкий Alpine/vanilla: 1-й клик = `von`, 2-й = `bis` (если ≥ von+min_nights
  и весь диапазон свободен — `range_available` уже гарантирует на сабмите), подсветка
  промежутка; затем проставляем в существующие hidden-инпуты формы дат и сабмитим GET
  (живёт рядом, переиспускает `_quote`/тарифы — ничего нового в расчётах).
- Без внешних библиотек (GDPR/no-SPA).

### 3.4 Интеграция в страницу номера
- В `stay_detail.html` рядом с формой выбора дат — блок календаря (грузится сразу
  server-side для текущего месяца; перелистывание — hx-get). На узких экранах —
  сворачивается под «📅 Verfügbarkeit ansehen».
- Виджет-embed (`?embed=1`, G10) — календарь должен работать и там (тот же партиал).

## 4. Демо
- Засеять у HOTEL (и pranasy-пансиона, если есть stays) несколько броней/блоков на
  текущий+следующий месяц в `demo_kits` (seed_records ветка stays), чтобы календарь
  показывал реалистичную смесь свободно/занято. Несколько примеров на номер.

## 5. Тесты
- `availability.month_availability`: занятость считается верно (бронь на 3 ночи → эти
  дни заняты; `quantity=2` + одна бронь → free=1 → is_free=True; блок → занято; прошлый
  день → in_past).
- Вьюха `unterkunft_unit_calendar`: 200 + сетка дней; гейт 404 без модуля stays;
  перелистывание месяца (`?year/&month`) рендерит соседний месяц.
- Шаблон: занятый день некликабелен (нет `data-date`), свободный — есть; «N frei» при
  quantity>1.
- Регрессия: `unterkunft_unit` по-прежнему отдаёт форму/`_quote` без изменений.

## 6. Инкременты (по одному = ветка/коммит → CI → мерж)
1. **C1 — данные:** `availability.month_availability` (+юнит-тесты occupancy). Без UI.
2. **C2 — вьюха+партиал:** эндпоинт `…/kalender/` + `_stay_calendar.html` (server-render
   текущего месяца, кнопки ‹ › hx-get). Тест вьюхи/гейта/перелистывания.
3. **C3 — выбор диапазона:** клик-2-даты → hidden-инпуты формы → сабмит (vanilla/Alpine),
   подсветка диапазона. Интеграция в `stay_detail.html` (+ сворачивание на мобильном).
4. **C4 — демо+embed:** брони/блоки в демо HOTEL; проверка работы в `?embed=1`.

## 7. Риски / на что смотреть
- **Производительность:** один запрос на месяц (не на день). При перелистывании —
  один hx-get. Не тянуть весь год.
- **Часовые пояса / «сегодня»:** считать `today` через `timezone.localdate()` (как RV3).
- **min_nights / сезоны:** клик-выбор не должен дублировать расчёт — финальную валидацию
  оставляем `_quote`/`range_available` на сабмите (календарь = подсказка, не источник истины).
- **anti-overbook:** не ослаблять — бронь по-прежнему через `book_stay` под блокировкой.
- **i18n/локаль:** недели Mo–So, даты в de-формате; строки через `{% trans %}`/`{% comment %}`
  (многострочные комментарии — только `{% comment %}`, урок CLAUDE.md).
