# ANTI-BITRIX — конструктор сайта и онбординг (сквозной анализ)

> Статус: **АНАЛИЗ / ЧЕРНОВИК на согласование** (2026-06-25). Конвенция: docs до кода.
> Source of truth по сделанному — `build-log.md`; план M20 — `m20-site-builder-plan.md`,
> `m20-retreat-pages-plan.md`. Этот документ — честная карта текущего билдера +
> рыночный бенчмарк + блюпринт «анти-Битрикс» (онбординг ≤10 шагов, блочная модель,
> on-canvas редактирование). Кодим только после подтверждения порядка.

Северная звезда владельца: **«Сделать анти-Битрикс: настройка за МАКС 5–10 шагов;
редактирование/визуализация страниц — нативно и максимально легко; чтобы „ребёнок“
мог собрать или отредактировать себе сайт. Возможно, какая-то блочная структура».**

Рамка: HTMX/Alpine/Tailwind, server-rendered, django-tenants, секции на JSON поверх
`Tenant.site_config` (без пер-секционных моделей). Это **эволюция, не переписывание
на SPA**. Все предложения ниже укладываются в этот стек.

## Current builder & onboarding (honest map)

### Онбординг сегодня: сколько шагов от регистрации до живого сайта

Поток (файлы: `apps/tenants/views.py`, `apps/tenants/services.py`, `apps/tenants/tasks.py`,
`apps/core/views.py`, `apps/tenants/onboarding.py`):

1. **Форма регистрации** (`BusinessSignupView`, `apps/tenants/forms.py::BusinessSignupForm`,
   шаблон `templates/tenants/onboarding.html`): 7 полей — `business_name`, `slug`
   (субдомен), `business_type` (10 типов), `city`, `email`, `password1/2`.
   На POST — `start_business_provisioning` (`services.py`): создаётся `Tenant` в public,
   `subscription_status="trial"`, `disabled_modules` предвыбираются по `business_type`
   (`modules.default_disabled_for`), создаётся Domain `{slug}.siteadaptor.de`, схема
   откладывается в Celery.
2. **Экран ожидания** (`signup_waiting`, `templates/tenants/provisioning.html`):
   спиннер + meta-refresh каждые 4 c; в фоне `provision_business` (`tasks.py`) создаёт
   схему, мигрирует TENANT_APPS, создаёт User+Membership(OWNER), шлёт письмо со
   ссылкой на логин. **Автологина нет.**
3. **Логин** (`/accounts/login/`, allauth) → `/dashboard/`.
4–8. **Onboarding-Wizard** `/dashboard/setup/` (`apps/core/views.py::setup_view`,
   `apps/tenants/onboarding.py`, шаблон `templates/tenant/setup.html`), 5 шагов:
   - **1. Was machst du?** — радио `business_type` (пере-предвыбор модулей).
   - **2. Was willst du anbieten?** — чекбоксы опциональных модулей; рекомендованные
     типу — сверху, прочие — под «Weitere Bausteine» с предупреждением.
   - **3. Die Basics** — адрес, часы, телефон, email (поля Tenant).
   - **4. Dein erster Inhalt** — ссылки на пресеты акций (`promotions/presets.py`) и на
     `/catalog/create`; **на самом шаге форма не сабмитится** — уход в другие вьюхи.
   - **5. Geschafft 🎉** — completed=true.

**Честный подсчёт.** Сайт становится **живым и публичным сразу после провижининга**
(шаг 2–3), даже с пустыми секциями — отдельного «Publish» нет. Но реалистичный путь
неподготовленного владельца до *наполненного* сайта — **~10 экранов**: форма → ожидание →
логин → 5 шагов мастера → возврат в билдер за hero/цветом/демо. Мастер целиком
**опционален и пропускаем**.

**Где застревает не-технарь:**
- **Нет AI/авто-генерации контента.** Мастер только тумблерит модули и собирает
  контакты — тексты hero/about/каталог владелец пишет с нуля. Пустой сайт после мастера.
- **Нет выбора дизайна на онбординге.** Тема-пикер (5 шаблонов, `sitetemplates.py`)
  живёт ОТДЕЛЬНО, постфактум, на `/dashboard/site/`. Владелец не видит «как будет
  выглядеть» во время настройки.
- **Слаг/субдомен** — абстракция, которую микробизнес не всегда понимает.
- **Шаг 4 — тупик внимания:** уводит в другую вьюху создавать акцию, легко «потеряться».
- **Демо-контент** (`apps/tenants/demo.py`, `load_demo`) — мощная штука (8–10 товаров,
  3 акции, услуги/номера/события по модулям, идемпотентно, обратимо через `clear_demo`),
  но **запускается кнопкой на `/dashboard/site/`, а не предлагается в мастере.**

### Билдер сегодня: что умеет и не умеет

Билдер — **серверный, форменный конфигуратор блоков** (НЕ drag-drop канвас). Вьюхи в
`apps/core/views.py`, маршруты `config/urls_tenant.py:55-62`, шаблоны `templates/tenant/site*.html`.
Разнесён по **6 экранам**:

| Экран | Вьюха / URL | Что делает |
|---|---|---|
| Site | `site_view` `/dashboard/site/` | hero/about-тексты, акцент, шрифт, галерея фото, **галерея шаблонов**, демо-контент |
| Home builder | `home_builder_view` `/dashboard/site/home/` | порядок/видимость секций, раскладки-пресеты, лимиты, тизеры архетипов, дизайн, **live-preview** |
| Menu | `menu_builder_view` `/dashboard/site/menu/` | верхнее/нижнее меню, 2 уровня, типы узлов |
| Sections | `sections_view` `/dashboard/site/sections/` | обложки архетипов: интро+баннер+галерея на раздел |
| Pages | `pages_view` `/dashboard/site/pages/` | пер-страничные раскладки (каталог/номера/события/похожие) |
| Preview | `site_preview` `/dashboard/site/preview/` | iframe + переключатель Desktop/Tablet/Mobile |

**Может:**
- Тумблерить ~17 готовых секций, менять порядок (нативный HTML5 drag в `site_home.html`
  + числовые поля; up/down в меню).
- Раскладки-пресеты на секцию-сетку (`siteconfig.LAYOUT_PRESETS`: list/cols2/3/4/gallery)
  + ручные override (cols/mobile/gap/width). Purge-safe Tailwind (`grid_class_string`).
- Тексты hero/about/CTA; пары FAQ/Testimonials/Process/Team через textarea «A | B».
- Дизайн: системные шрифты (`FONTS`, без веб-шрифтов — GDPR), акцент-цвет, hero plain/accent.
- **Live-preview (V1):** форма дебаунсит 400 мс → POST `site_preview_draft` (черновик в
  `request.session`) → iframe `?preview=1` рендерит черновик (`promotions/public_views.py::storefront_home`).
- **Инлайн-правка текста** в превью: элементы `[data-edit]` становятся `contenteditable`
  → blur POST `site_inline_edit` (whitelist `siteconfig.TEXT_FIELDS` + `NESTED_TEXT_FIELDS`).
- Галерея шаблонов (`sitetemplates.apply_template`) — в один клик раскладка+тексты+акцент,
  без затирания непустых текстов владельца.
- Демо-контент идемпотентно/обратимо.

**Не может (ключевые пробелы к северной звезде):**
- ❌ **Нет on-canvas редактирования** кроме текста: видимость/порядок/раскладку нельзя
  менять кликом по самому превью. Drag — только в форме слева, не на канвасе.
- ❌ **Нет инсертера «+»** / библиотеки блоков с превью-эскизами. Секции фиксированы —
  только вкл/выкл, нельзя «вставить ещё один блок текста/картинку».
- ❌ **Нет Undo/Redo** и истории версий — Save коммитит в БД сразу.
- ❌ **Нет загрузчика картинок в hero/баннеры** — только ручной ввод URL (медиа-библиотека
  есть лишь на отдельных страницах галереи).
- ❌ **Редактирование размазано по 6 экранам** — владелец переключает контекст.
- ❌ **Нет AI-генерации** текстов/наполнения.
- ❌ **Диспетчер секций — хардкод if/elif** в `templates/storefront/home.html` (18 веток),
  не реестр; добавление секции = правка шаблона + вьюхи. Мешает динамике/«+».

**Клики до типовой задачи (не-технарь):** сменить hero-заголовок — 4 клика (Site →
раскрыть → поле → Save) либо клик-в-превью (есть); сменить hero-картинку — 4 клика +
знать URL; переставить секции — drag в форме + Save; включить секцию — 2 клика; добавить
произвольный блок — **невозможно**.

## Market benchmark (what makes builders child-simple)

Что делает «детские» билдеры простыми (и чего избегает Битрикс):

**1. AI/авто-онбординг вместо пустого холста.**
- **Jimdo Dolphin** — мастер собирает *реальный контекст бизнеса* (цель, категория,
  стиль, цвета) и **тянет данные из Google/Facebook** (адрес, телефон, фото, товары),
  выдавая **готовый черновик за <3 мин**, потом — два варианта на выбор
  ([Jimdo](https://help.jimdo-dolphin.com/hc/en-us), [99designs](https://99designs.com/blog/web-digital/jimdo-website-how-to/)).
- **Durable** — «сайт за 30 секунд» из 3 ответов (тип/название/локация): копирайт,
  картинки, контакт-форма, SEO ([Durable](https://durable.com/ai-website-builder)).
- **Hostinger AI** — мультистраничный сайт <60 c из простых вопросов
  ([Hostinger](https://www.hostinger.com/ai-website-builder)).
- Урок: **ноль пустых полей.** Онбординг = «ответь на пару вопросов → получи
  наполненный сайт → правь», а не «вот пустой каркас, заполняй сам».

**2. Section-first, не pixel-first (ADI-модель).**
- **Wix ADI** строит из **целых секций**: правка = двигать секции и кликать по блоку
  («edit content / different preset design / change color»), **без drag-drop по пикселям**.
  Wix Editor (pixel-grid drag) — мощнее, но сложнее и легко «сломать»
  ([tech.co](https://tech.co/website-builders/wix-editor-vs-wix-adi), [wpshout](https://wpshout.com/wix-adi-vs-wix-editor/)).
- **Squarespace** держит **строгие дизайн-гардрейлы** — «эстетический baseline» для
  не-дизайнеров; качество держится автоматически ([sitebuilderreport](https://www.sitebuilderreport.com/)).
- **Carrd** — намеренная узость (одна страница, минимум опций) = скорость и отсутствие
  bloat ([sitebuilderreport](https://www.sitebuilderreport.com/squarespace-alternatives)).
- Урок для нас: наша секционная JSON-модель **уже ADI-подобна** — это сильная сторона,
  её надо доводить, а не менять на pixel-grid.

**3. Гардрейлы «нельзя сделать уродливо».**
- Squarespace **не показывает** опции, которыми можно сломать дизайн (нет произвольных
  шрифтов/раскладок там, где они навредят); Divi даёт свободу, но «через месяцы
  консистентность деградирует» ([Elementor](https://elementor.com/blog/no-code/),
  [FatLab](https://fatlabwebsupport.com/blog/wordpress-development/divi-review/)).
- Урок: **ограниченный набор пресетов + темы-токены** (наши `FONTS`, акцент, layout-пресеты)
  — это фича, а не баг. Никаких произвольных CSS/отступов в руках владельца.

**4. On-canvas: кликни по элементу → правь на месте.**
- Wix ADI / Squarespace Fluid Engine: клик по блоку → попап-настройки прямо на канвасе;
  **отдельная мобильная правка** (Fluid Engine правит mobile независимо)
  ([Squarespace](https://support.squarespace.com/hc/en-us/articles/6421525446541)).
- Урок: у нас уже есть `[data-edit]` contenteditable — это **зерно** on-canvas; нужно
  расширить до «клик по секции → попап настроек», «+ между секциями».

**5. Анти-паттерн Битрикс.** Жалобы: «overwhelming», «cluttered», «легко потеряться»,
«пытается делать всё» — слишком много опций/настроек на экране, нет фокуса
([HypeGig](https://hypegig.com/bitrix24-review/), [Capterra](https://www.capterra.com/p/113540/Bitrix24/reviews/)).
Урок: **меньше экранов, прогрессивное раскрытие, один смысл на экран, дефолты-которые-работают.**

## Gap to north star

**Насколько близко (честно): ~55–60%.** Архитектурный фундамент под «анти-Битрикс»
**уже правильный** и местами обгоняет конкурентов: секционная JSON-модель (ADI-подобна),
live-preview с draft-в-сессии, инлайн-правка текста, layout-пресеты-гардрейлы, темы без
веб-шрифтов, демо-контент идемпотентный, реестр архетипов (`primary_item`/`purchase_mode`).

**Три самых больших разрыва:**
1. **Онбординг даёт пустой сайт.** Конкуренты выдают наполненный черновик за <60 c;
   мы — каркас + ручной труд. Это **главный рычаг** (см. ниже).
2. **Редактирование не «нативное на канвасе».** On-canvas есть только для текста; всё
   остальное — формы на 6 экранах. «Ребёнок» не соберёт интуитивно — не видит, *куда*
   кликнуть на самой странице.
3. **Блоки не добавляются.** Нет «+» и библиотеки блоков; фиксированный набор секций
   нельзя расширить (второй текст-блок, баннер-картинка, кнопка) — мешает свободе
   «собрать себе».

**Единственный самый большой рычаг:** **онбординг, который выдаёт уже наполненный,
красивый сайт за ≤10 шагов** (тема + демо-контент + контакты, авто-подставленные),
так чтобы первый экран после мастера был «вот твой готовый сайт, поправь что хочешь»,
а не «вот пустой каркас». Это сразу закрывает разрыв №1 и психологически снимает страх
«пустой страницы». У нас **уже есть кирпичи** (`sitetemplates`, `demo.load_demo`,
`presets`) — их нужно сшить в один линейный мастер.

## ≤10-step onboarding blueprint

Цель: от регистрации до **наполненного, опубликованного, красивого** сайта за ≤10 шагов,
каждый — одно решение, дефолт всегда осмысленный, «Weiter» без ввода допустим. Линейный,
HTMX-частями (один URL `/willkommen/`, шаги — hx-swap внутреннего фрагмента; состояние —
в `site_config["onboarding"]`, как сейчас).

| # | Шаг | Что спрашиваем | Что под капотом (используем существующее) |
|---|---|---|---|
| 1 | **Регистрация** | название, email, пароль (слаг — авто из названия, правится позже) | `BusinessSignupForm` упростить: убрать `business_type`/`city`/`slug`-ручной; слаг = `slugify(name)` |
| 2 | **Was machst du?** | тип бизнеса (плитки с иконками, не радио) | `business_type` → `default_disabled_for` (модули предвыбраны) |
| 3 | **Stil & Farbe** | выбор из 5 шаблонов как **визуальные превью-карточки** + акцент-цвет | `sitetemplates.TEMPLATES` (показать миниатюру каждого), `apply_template` |
| 4 | **Deine Daten** | адрес/часы/телефон (+ «aus Google holen?» — фоллбэк ручной) | поля Tenant; *(опц. позже: импорт из Google Places)* |
| 5 | **Beispiel-Inhalte?** | «Naполнить демо-товарами/акциями, чтобы сразу видеть сайт?» (Ja/Nein) | `demo.load_demo(tenant)` — **предложить прямо здесь**, обратимо `clear_demo` |
| 6 | **Provisioning** (фон) | спиннер; параллельно создаётся схема | `provision_business` (как сейчас, но шаг встроен в мастер) |
| 7 | **Dein Banner** | заголовок + подзаголовок hero (предзаполнены из шаблона) + загрузка фото | hero_title/text + **новый загрузчик в hero** (`catalog.images.save_product_image`) |
| 8 | **Vorschau** | «Так будет выглядеть» — live-preview на весь экран, mobile/desktop | существующий iframe `?preview=1` |
| 9 | **Fertig 🎉 / Veröffentlichen** | «Sайт онлайн на `name.siteadaptor.de`» + кнопка «открыть» | сайт уже публичен; кнопка-«поделиться» |
| 10 | *(опц.)* **Nächste Schritte** | 3 карточки: «добавить товар», «настроить меню», «пригласить коллегу» | дип-линки в кабинет |

Шаги 6 — фоновый (не считается «работой» владельца). Реалистично владелец делает
**7–8 осмысленных решений** и видит **наполненный** сайт. Демо-контент (шаг 5) — ключ:
после мастера сайт не пустой.

Ключевые отличия от сегодня: (а) тема-пикер и демо **внутри** мастера, не постфактум;
(б) hero-картинка грузится файлом, не URL; (в) финал = «вот твой готовый сайт», не «вот
каркас»; (г) один линейный URL вместо «логин → дашборд → setup».

## Block model proposal

Принцип владельца: **«архетип = главный товар + способ покупки»** уже закодирован
(`apps/core/archetypes.py`: `PRIMARY_SECTION`, `PURCHASE_MODE`, `purchase_label`). Блочную
модель строим **поверх той же JSON-схемы** (`site_config["sections"]`), без новых моделей.

### Таксономия блоков (две категории + один новый класс)

**A. Functional / archetype-блоки** (data-driven, гейтятся активным модулем) —
*уже есть, не трогаем семантику:* `promotions`, `products`, `categories`, `events`,
`stay_search`, `stay_rooms`, `archetypes`, `contact`. Это «главный товар» архетипа.

**B. Content / marketing-блоки** (текст из `site_config`) — *уже есть:* `hero`, `about`,
`process`, `team`, `cta`, `testimonials`, `trust`, `faq`, `gallery`.

**C. НОВОЕ — свободные «простые блоки»** (то, чего просит владелец для «собрать себе»):
- `text` — заголовок + абзац (rich-lite: bold/ссылка).
- `image` — одна картинка + подпись.
- `image_text` — картинка слева/справа + текст (раскладка-токен).
- `button` — CTA-кнопка (label + target из реестра меню: archetype/category/url/anchor).
- `spacer`/`divider` — воздух.
- `embed` — GDPR-дружелюбный (карта/видео 2-Klick, как `gallery_video`).

Класс C решает «нет инсертера/нельзя добавить ещё блок»: эти блоки **множественные**
(в отличие от текущих single-секций), поэтому секция получает `id` (uuid) помимо `key`:
`{"key":"text","id":"...","data":{...},"layout":{...},"enabled":true}`. `normalize`
расширяется: known single-секции — как сейчас; `key in REPEATABLE_BLOCKS` → массив с `id`.

### Как блоки добавляются/удаляются/переставляются/правятся

- **Реестр вместо хардкода.** Заменить if/elif в `templates/storefront/home.html` на
  **inclusion-tag-диспетчер** `{% render_block b %}` с картой `BLOCK_TEMPLATES = {key:
  partial}` (single source of truth). Это разблокирует динамику и «+». Эволюция, не
  переписывание — партиалы те же.
- **Добавление:** инсертер «+» (см. ниже) → POST добавляет запись в `sections` → hx-swap
  превью. Для C-блоков — с новым `id`.
- **Удаление/скрытие:** `enabled:false` (как сейчас) или удаление записи (для C).
- **Перестановка:** drag (как сейчас) — но и **на канвасе** (см. ниже).
- **Правка:** клик → попап настроек блока (текст, раскладка-пресет, лимит, источник).

### Гардрейлы «нельзя сделать уродливо»

- Раскладки — **только пресеты** (`LAYOUT_PRESETS`), «Дополнительно» (ручные cols/gap)
  свёрнуто. Уже так.
- Цвета — **только акцент-токен** + plain/accent hero; никаких произвольных CSS.
- Шрифты — **3 системных стека** (`FONTS`); без веб-шрифтов (GDPR + нельзя «сломать»).
- C-блоки рендерятся в тех же `max-w`/spacing-токенах — владелец не задаёт пиксели.
- Капы (`_MAX_HEROES`, `_MAX_ITEMS`, `_MAX_GALLERY`) — анти-флуд, оставить.

## On-canvas native editing

Цель: «ребёнок кликает по самой странице и правит». Прагматично на HTMX/Alpine, поверх
уже работающего live-preview (`site_preview_draft` + `[data-edit]` contenteditable).

**Эволюция превью-iframe в «редактор-канвас» (режим `?edit=1`):**

1. **Click-to-edit на всё, не только текст.** Каждый блок в превью оборачиваем
   `data-block="<key|id>"`. Клик по блоку → Alpine ловит `postMessage` от iframe →
   родитель открывает **попап настроек** этого блока (hx-get партиал формы) сбоку/снизу.
   Текст по-прежнему contenteditable inline (`[data-edit]` — уже есть).
2. **Инсертер «+».** Между блоками в edit-режиме — тонкая зона с «+»; клик → выезжает
   **библиотека блоков с миниатюрами** (статичные SVG/скрин-эскизы пресетов). Выбор →
   hx-post в `site_preview_draft` (добавить запись) → hx-swap фрагмента превью. Гардрейл:
   список = только из реестра, всё заведомо «красивое».
3. **Перетаскивание на канвасе.** SortableJS на контейнере секций в iframe; `onEnd` →
   POST нового порядка в draft → swap. Дублирует drag-в-форме, но «на месте».
4. **Mobile/Desktop toggle.** Уже есть (ширина iframe). Добавить: правка остаётся общей
   (не делаем независимый mobile-контент — это усложнение; гардрейл: пресеты адаптивны
   сами, `grid_class_string` уже даёт mobile-колонки).
5. **Undo/Redo (lightweight).** Draft в сессии — добавить **стек снимков** (последние
   N=20 JSON-состояний в `session["site_preview_history"]`); Ctrl+Z / кнопка → POST
   «откатить» → swap. Дёшево, без моделей. На Save — коммит текущего draft в `site_config`.
6. **Один экран вместо шести.** Свести Site/Home/Sections/Pages в **один builder-экран**
   с левой панелью-«деревом блоков» + канвас справа; Menu отдельно (это навигация).
   Преимущество: владелец не теряется (анти-Битрикс).

**Что переиспользуем:** `site_preview_draft` (черновик), `parse_content_sections`
(один парсер для формы/билдера/draft), `normalize_layout`/`grid_class_string`
(purge-safe), `[data-edit]`/`site_inline_edit`. Никакого React — Alpine-компонент-обёртка
+ HTMX-партиалы + iframe↔parent `postMessage`.
