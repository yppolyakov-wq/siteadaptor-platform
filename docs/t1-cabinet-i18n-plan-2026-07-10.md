# T1 — перевод кабинета (FB-12), план (2026-07-10)

Задача владельца: админ-панель на всех языках (не только DE), переводами управлять.
Решение владельца: T1-a (инженерия) → T1-b (перевод DeepL free, сначала EN) → T1-c
(django-rosetta для правки переводов). Объём T1-a: **сначала ключевые экраны**, потом
добить. Замок: кабинет в DE = как раньше (msgid), тесты зелёные.

## Ключевая проблема (почему не просто «включить перевод»)
Кабинет и витрина живут на ОДНОМ субдомене тенанта (`x.siteadaptor.de/` = витрина,
`x.siteadaptor.de/dashboard/` = кабинет) и делят Django-cookie языка
(`LANGUAGE_COOKIE_NAME`, читает `LocaleMiddleware`). Значит язык кабинета нельзя
хранить в той же куке — он столкнётся с языком витрины, который выбирает КЛИЕНТ
(`set_language`, валидируется по `tenant.active_locales`). Нужен ОТДЕЛЬНЫЙ механизм
локали кабинета, перекрывающий язык для кабинет-путей.

## T1-a — механизм (этот инкремент)
1. **`settings.CABINET_LANGUAGES`** — курируемый список переведённых языков кабинета
   (старт `("de", "en")`; растёт в T1-b по мере готовности `.po`). Не путать с
   `settings.LANGUAGES` (весь реестр витрины).
2. **`apps/core/i18n_cabinet.py`** — `cabinet_locales()` (список + лейблы) и
   `resolve_cabinet_locale(request)`: сессия `cabinet_lang` (если ∈ CABINET_LANGUAGES)
   → иначе `settings.LANGUAGE_CODE` (**de по умолчанию — кабинет как раньше**).
   НЕ привязываем к `tenant.default_locale` (то — язык витрины, другое понятие).
3. **`CabinetLocaleMiddleware`** (после `LocaleMiddleware` в MIDDLEWARE): для путей
   `("/dashboard/","/catalog/","/promotions/","/imports/","/crm/")` — `translation.activate(loc)`
   + `request.LANGUAGE_CODE = loc`. Перекрывает язык только в кабинете; витрина не
   затронута.
4. **`set_cabinet_lang`-вьюха** (+ роут в urls_tenant) — пишет `session["cabinet_lang"]`
   (валидация по CABINET_LANGUAGES), redirect назад (referer). GET/POST.
5. **Контрол в шапке** `_base_dashboard.html` — компактный переключатель языка КАБИНЕТА
   (иконка 🖥/globe + текущий язык), отдельно от «🌐 Sprachen» (та — про язык ВИТРИНЫ).
   Показываем только при `len(cabinet_locales()) > 1`. Контекст-процессор даёт
   `cabinet_lang`/`cabinet_langs`.
6. **Пилот**: перевести 2–3 ВИДИМЫЕ строки шапки в `locale/en/.po`, чтобы переключение
   на EN было видно (доказательство end-to-end). Систематический перевод — T1-b.
7. **Замки/тесты**: switcher пишет/валидирует сессию; resolve = de без сессии, override
   сессией; middleware активирует на кабинет-пути и НЕ трогает витрину; при `cabinet_lang=en`
   пилотная строка рендерится EN (msgfmt-компиляция, как email_i18n); DE-рендер без сессии
   не изменился (существующие тесты зелёные).

## T1-a.2 — добить обёртку строк (следующий инкремент)
Аудит ключевых экранов (дашборд/Sortiment/Verkäufe+доска/Einstellungen) на необёрнутые
строки → `{% trans %}`/`gettext`. Каждая пачка гейтится (DE msgid не изменился).

## T1-b — перевод (дёшево)
`makemessages -l en` (и далее по CABINET_LANGUAGES) → `locale/<lang>/.po` →
**DeepL free API** (500k/мес; открыть `api-free.deepl.com` + ключ) для массового перевода
→ LLM/ручная вычитка ~30–50 хитрых строк. `.mo` компилируется в CI/deploy (как L4).
Языки после EN — по решению владельца (реестр: tr/ru/uk/pl/fr/it/es/nl/pt).

## T1-c — управление переводами
`django-rosetta` — веб-редактор `.po` в кабинете (владелец правит формулировки без кода).
Гейт по роли/суперюзеру. Опц. per-tenant оверлей правимых лейблов — если нужна правка
без пересборки `.mo`.

## Разведённые понятия (важно для владельца, чтобы не путать)
- **Язык витрины** (что говорит сайт клиенту) — `/dashboard/settings/languages/`, есть.
- **Переводы контента** (названия/описания товаров/услуг) — свитчер в форме (Ф1), есть.
- **Язык кабинета** (меню/кнопки админки) — T1, строится здесь.
