## 1. Философия и принципы

1. **Видение целиком сохраняем.** Цель — универсальная **Business OS для SMB** (CRM +
   ERP + Маркетплейс + Dropshipping + Booking + Заказы + Бухгалтерия + Workflow + AI +
   мультиканальные продажи + конструктор сайтов). Полный список — vision §2.
2. **Микробизнес получает полный инструмент из коробки — в первую очередь.** Каждый
   архетип (A1–A9) должен работать на 100% дневного цикла **по умолчанию**: нужный
   набор модулей предвыбран по типу бизнеса (реестр `apps/core/modules.py`
   `recommended_for`/`suited_for` + онбординг-визард), без ручной сборки.
3. **Строим легко поверх готовых движков, а не тяжёлый ERP с нуля.** У нас уже есть
   `catalog`, `orders`, `booking`, `stays`, `jobs`, `finance`, `crm`, `aggregator`,
   `billing`, `publishing`, `notifications`. «ERP/маркетплейс/AI» = тонкие надстройки
   над этим ядром (ledger поверх остатка, цепочки поверх Order, и т.д.), а не вторая
   система. Мы **уже сделали больше**, чем планировалось к этой точке (см. §4).
4. **Архитектурные швы под миграции закладываем заранее** (см. §7), чтобы будущие
   фазы были **аддитивными** (новые поля/модели), а не переписыванием с болезненным
   backfill.
5. **Инварианты платформы (не нарушаем):** multi-tenant (schema-на-тенанта),
   модульность (реестр + гейтинг), DSGVO-first (без трекинг-куки, Double-Opt-In,
   право на стирание), идемпотентность побочек (FSM + dedupe), региональное хранение
   (`Tenant.data_region` заложен).
6. **Монетизация-инвариант:** базовый тариф ~39 €/мес **без комиссий с оборота**
   (вариант B: плата за пользование строкой в счёте, не удержание в платеже клиента).
   Комиссионная маркетплейс-модель — отдельная развилка (§8), не отменяет инвариант.

### Сквозной модуль M22 — Чат, поддержка и тикеты (✅ a+b+c в `main`, v1 async)
Поглощает и расширяет отложенный пункт «чат покупатель↔бизнес». Единый движок
коммуникаций в **двух scope**:
- **M22a — Чат клиент↔бизнес (TENANT):** виджет «вопрос по товару/акции/брони» на
  витрине + единый inbox в кабинете владельца; тред привязан к `Customer` (виден в
  CRM 360°) и к объекту (товар/акция/заказ/бронь). Это **инструмент из коробки** для
  всех архетипов A1–A9. → product, Stage 1–2.
- **M22b — Helpdesk-тикеты бизнеса (TENANT):** тот же тред со статусом
  (open→pending→resolved→closed), приоритетом, назначением на сотрудника, SLA — это
  «система поддержки в виде чата с тикетами», которую **наши тенанты дают своим
  клиентам**. → product, Stage 2.
- **M22c — Платформенная техподдержка (SHARED):** владелец тенанта пишет в
  SiteAdaptor-поддержку из кабинета (чат+тикет); платформа видит тикеты по всем
  тенантам. Наша ops-/онбординг-нужда + догфудинг M22a/b. → ops, может стартовать
  рано (Stage 0–1, помогает пилоту).

**Версионность:** v1 — асинхронные треды (HTMX-поллинг + доставка/нотификации через
`apps.notifications`: email-тред, WhatsApp deep-link); **realtime (websocket/Channels)
и AI-триаж — Stage 3, аддитивно** (модель не переписываем). Модель/швы — §7.

### Сквозной модуль M23 — Маркетинг и медийное продвижение (соцсети/продажи/реклама)
Расширяет **M3 Publishing** (готовый фреймворк `Channel`/`Publication`/SM + адаптеры,
B1 GBP) до полного маркетинг-контура. Прямо отвечает на «каналы продаж/интеграций/
публикаций». Три грани:
- **M23a — Органическое распространение (соц-постинг):** ✅ **v1 в `main`**

  container → publish) на фреймворке `apps.publishing`; хук `PromotionSM` «акция
  активна → пост, завершилась → снятие» уже работал (B1 GBP); дедуп даёт
  `Publication` SM (одна публикация на канал). Токены per-канал в кабинете
  (`docs/meta-social-setup.md`); боевое — после Meta App Review. Дальше (Stage 2):
  расписание/шаблоны/превью, TikTok/X. → P2.9 «каналы публикаций», приоритет владельца.
- **M23b — Social commerce / продажи через соцсети:** каталог-фид (Meta Commerce
  Manager / Google Merchant Center) из `catalog`, product-tagging, синхронизация
  наличия (R3)/цен; «buyable»-посты ведут на нашу витрину/корзину (checkout есть).
  → **Stage 2** (часть M21 Интеграции).
- **M23c — Управление медийным продвижением (реклама):** `Campaign` (цель/бюджет/

  `AdInsight` (охват/клики/конверсии/ROAS) с привязкой к акции/листингу. Атрибуцию
  НЕ изобретаем — `source_channel`/`?ch=`/QR уже есть. → **Stage 2–3** (тяжелее: OAuth+API).

Переиспуем: `publishing` (каналы/Publication SM), `catalog` (фид), `promotions`
(хук+атрибуция), `aggregator` (листинги), OAuth-токены per-канал (паттерн GBP). v1 —
органика+фиды; платная реклама — следом. Швы — §7, развилки (трекинг/монетизация) — §8.

## 7. Архитектурные швы под будущие миграции (закладываем заранее)

Принцип: закладываем **сейчас** только те швы, что (а) дёшевы, (б) болезненны для
backfill позже, (в) прямо названы в видении как ранние хуки. Остальное — аддитивно
**вместе с модулем** (django-миграции дешёвы; спекулятивные пустые поля не плодим).

**Заложить сейчас (отдельные узкие миграции, без логики):**
- ✅ **Сделано (`8bfd3ae`):** `Order.parent_order` (self-FK, SET_NULL) +
  `Order.supplier_tenant_schema` (char) — пассивные хуки dropshipping/маркетплейс
  (M11→M14/M15). Миграция orders/0005.
- ✅ **Сделано (`8bfd3ae`):** роли — `apps.core.Membership(user OneToOne, role
  owner/admin/staff)` (TENANT-scope) + `apps.core.roles.role_of()`/`is_owner()`
  (дефолт owner → легаси без backfill); владелец получает owner при провижининге.
  Миграция core/0001. Гейтинг во вьюхах НЕ включён (точка централизации на будущее).
- `Tenant.data_region` — **уже есть** (P2.10), используем при многорегионе.

**Закладывать вместе с модулем (НЕ сейчас — иначе мёртвая схема):**
- `StockMovement` (M10): ledger поверх R3 (`product/variant, delta, reason, ref_*`) —
  вводим при старте полноценного склада; текущий атомарный остаток остаётся источником.
- `FinancialMovement` (M13): обобщение `RevenueEntry` до доход/расход/коррекция +
  `Account` — при расширении финмодуля; миграция данных из RevenueEntry — аддитивна.
- **Шина доменных событий / outbox** (M15 dropshipping-синк, M19 workflow-триггеры) —
  вводим при первом из этих модулей; сейчас побочки и так идемпотентны (FSM+dedupe).
- Channel-адаптеры (M3/M21) — фреймворк уже есть, просто добавляем реализации.
- **M22 Чат/поддержка** — вводим со своим модулем (новое app, напр. `apps.inbox`):
  `Conversation` (scope customer↔business | tenant↔platform; subject; для тикета —
  status-FSM/priority/assignee/SLA; FK `Customer`/`PortalUser`; `channel`/`source`:
  web-widget/email/whatsapp; `last_message_at`) + `Message` (author: staff/customer/
  system; body; вложения; read-флаги). **TENANT-scope** для M22a/b, **SHARED-scope**
  для M22c (как billing). Швы под будущее заложить сразу в первой миграции, пустыми:
  `realtime`-канал и `ai_handled`-флаг (под Stage 3), `external_ref` (под email/WA-тред).
  Переиспуем: `notifications` (доставка), `Customer`/`PortalUser` (стороны), `core.fsm`
  (статусы тикета), dedupe (без дублей писем). DSGVO: тело сообщений = PII →
  экспорт/стирание + ретеншн.
- **M23 Маркетинг/реклама** — строим на готовом `apps.publishing` (Channel/Publication/
  SM): соц-адаптеры = новые типы Channel (organic), как B1 GBP. Новое со своим модулем:
  каталог-фид-экспорт (Meta/Google — как `sitemap.xml`), модели `Campaign`/`AdInsight`
  (TENANT) под платную рекламу. OAuth-токены per-канал — паттерн GBP refresh-token в
  конфиге. Атрибуция уже есть (`source_channel`/`?ch=`/QR) — не дублируем. Лимит DSGVO:
  ad-пиксели/трекинг на витрине конфликтуют с «без трекинг-куки» → §8.

## 8. Открытые развилки / журнал решений

- **Маркетплейс-монетизация (M14):** комиссия с оборота конфликтует с инвариантом «39 €
  без комиссий». Варианты: (B, текущий) плата за пользование строкой в счёте; (A)
  application_fee в платеже; (3) платформа собирает + payout (резерв под полноценный
  маркетплейс). Решать при старте M14.
- **Stripe Connect redirect-URI на субдоменах** (P2.5a) — единый callback на основном
  домене со state→tenant **или** wildcard. Блокер боевой онлайн-оплаты → выбрать в Stage 0.
- **Canonical-политика custom-доменов** — какой хост главный для SEO.
- **Heilberuf** (массаж/Physio): только запись, без медкарт (Heilmittelwerbegesetz).
- **Тарифные tier'ы** (Starter/Standard/Pro, add-ons) — при наборе функций per-module.
- **M22 realtime-транспорт:** v1 async (HTMX-поллинг + email/WA через `notifications`);
  realtime — Django Channels (websocket) или внешний (Pusher/Ably) — выбрать при Stage 3.
  Чат/тикеты как платный add-on к тарифу — кандидат при tier'ах.
- **M23 ad-трекинг vs «без трекинг-куки»:** Meta/Google Pixel на витрине нарушает
  DSGVO-инвариант. Варианты: server-side Conversions API (без куки), consent-gate, либо
  атрибуция только по нашим `?ch=`/QR без пикселя. Решить при M23c.
- **Монетизация медиа (M23):** управление рекламой/бюджетами — кандидат в платный
  add-on/tier; ad-spend идёт через рекламные кабинеты бизнеса (бюджеты НЕ эквайрим).
