# Редактор: живое применение изменений блока без «перезагрузки» (2026-07-12)

Запрос владельца: «при изменении в онлайн-редакторе страница обновляется — при
перетаскивании блока, изменении состояния (видимости) или положения. Можно ли
проводить изменения без перезагрузки страницы, прямо сразу в блоке».

## Диагноз (разведка Explore-агентом)

ЕДИНСТВЕННЫЙ механизм live-превью в `templates/tenant/site_home.html`:
`schedule()` (debounce 400 мс) → `push()` → **полная перезагрузка кадра**
`frame.contentWindow.location.replace(previewUrl())` (строка ~2222). Через него
идут ВСЕ изменения панели: текст, цвет, ширина, положение, видимость, порядок.

Почему правка ТЕКСТА кажется «живой», а три названных случая — «перезагрузкой»:
правка текста мутирует DOM кадра НА МЕСТЕ (`contenteditable`) ДО перезагрузки,
поэтому последующая тихая перезагрузка невидима. У перестановки/видимости/ширины
нет мутации на месте → изменение видно ТОЛЬКО после перезагрузки → «прыжок».

Диспозиция кадра: `#home-prev-frame` — `h-full`, внутренний скролл документа
(`frame.contentWindow.scrollY`), который `location.replace` сбрасывает в 0 →
страница «прыгает» к началу. Блоки в кадре помечены `data-sf-section="<id/key>"`
(и на главной, и на страницах). Отключённые секции в рендере ОТСУТСТВУЮТ
(`siteconfig.page_section_order`: `if e.get("enabled")`), в т.ч. в превью.

## Решение (тот же приём, что у текста: оптимистичная мутация + тихая сверка)

Не убираем перезагрузку (она — надёжная сверка с сервером). Делаем её НЕВИДИМОЙ и
добавляем мгновенную оптимистичную мутацию кадра там, где она точная:

1. **Сохранение прокрутки кадра** через тихую перезагрузку (универсально, все
   изменения). В `push()` перед `location.replace` запоминаем `scrollY`; в
   обработчике `load` кадра — возвращаем. Убирает «прыжок к началу» для ширины,
   положения, видимости, порядка, цвета — всего.

2. **Оптимистичная перестановка секций ГЛАВНОЙ** — на `drop` двигаем сам
   `[data-sf-section]` узел в кадре (простые соседи → точно), затем `moveBlock` +
   `schedule()`. Мгновенная перестановка; фоновая перезагрузка лишь сверяет.
   Страницы (C-блоки в row-группах `md:flex`) — БЕЗ оптимистичного переноса
   (структура рядов не воспроизводится точно), полагаемся на тихую перезагрузку
   со сохранённой прокруткой.

3. **Оптимистичная видимость** — чекбокс «Видимый» (`enabled_[cb_]<key>`) сразу
   прячет/показывает `[data-sf-section]` узел в кадре (`display`), фоновый
   `schedule()` сверяет. Показ работает, если узел ещё в DOM; иначе перезагрузка
   добавит его (прокрутка сохранена).

Ширина/положение (пересчёт классов + row-группировка на сервере) точно не
воспроизвести на клиенте — им достаточно п.1 (тихая перезагрузка без прыжка).

## Риск / откат

Все три пункта АДДИТИВНЫ: перезагрузка остаётся как источник истины. Худший
случай оптимистичной мутации — кратковременное расхождение, которое сверяющая
перезагрузка (через 400 мс, со сохранённой прокруткой) исправляет. Функциональной
регрессии нет; миграций нет.
