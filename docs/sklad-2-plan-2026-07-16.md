## 2. Три эпика (канон roadmap)

| # | Эпик | Суть | Для кого | Инвазивность |
|---|------|------|----------|--------------|
| **E1** | **Chargen / MHD** | Партия (`Lot`: product/variant, qty, MHD-дата, received_at, code) → приёмка по партиям · FEFO-списание · алерты «скоро истекает»/просрочено · связь с Verderb-причиной | еда (bakery/butcher/grocery/cafe/restaurant) — **ядро сегмента** | ⚠️ FEFO-на-продаже трогает атомик заказа |
| **E2** | **Мультисклад** | Локация как измерение остатка: `StockMovement += location`, баланс/реконсиляция per-location, Umlagerung (переброс) | мульти-Filiale (реже у микробизнеса) | ⚠️⚠️ ломает модель одного счётчика |
| **E3** | **Закупки M12** | `Lieferant` + `Bestellung`: Bestellvorschlag (T5) → черновик заказа → приёмка закрывает заказ и книжит Wareneingang; связать `cost_price` с реальными закупками | ритейл/еда с закупками | ⚠️ поверх приёмки; частично аддитивно |

**Рекомендация по порядку: E1 → E3 → E2.**
- **E1 первым:** максимальная ценность для целевого сегмента (флагманские демо —
  пекарня/мясная/кафе; MHD в DACH — практическая необходимость, не «ERP-роскошь»).
  Самодостаточен, аддитивен на ранних фазах.
- **E3 вторым:** естественное продолжение T5-Bestellvorschlag; приёмка E3 переиспользует
  приёмку-по-партиям из E1 (Wareneingang с MHD/Charge).
- **E2 последним:** мультисклад нишевой для микробизнеса и сильнее всего ломает
  инвариант счётчика — берём, только если будет спрос.

## 3. Ключевое архитектурное решение (общее для E1/E2)

**Партии и локации дробят остаток — счётчик `stock_quantity` больше не может быть
единственной истиной для разбивки.** Два подхода:

- **Вариант A (рекомендую): счётчик = ИТОГО, партии/локации — разбивка поверх, сверяемая
  реконсиляцией.** `Lot.qty_remaining` (и, для E2, per-location баланс из леджера)
  живут РЯДОМ; `Σ` должен сходиться со счётчиком — ровно как сегодня леджер сверяется
  со счётчиком. Инвариант D1 сохраняется, движки заказа не переписываются на раннем
  этапе, FEFO — отдельная опт-ин фаза. **Минимальный риск, инкрементально.**
- **Вариант B: полноценный per-(entity, location, lot) баланс-леджер как истина,
  счётчик — производное.** Честнее для «настоящего ERP», но это переписывание ядра
  остатка (все врезки + движки) — большой риск, оправдан только при реальном мульти-
  складе. **Не рекомендую на старте.**

Беру **Вариант A** по умолчанию (подтверди в §4).

## 4а. Решения владельца (2026-07-16) — ЗАФИКСИРОВАНЫ

- **Порядок: «все 3»** — делаем E1 → E3 → E2 (порядок §2 подтверждён; scope = все три эпика).
- **Глубина E1: сразу ПОЛНЫЙ FEFO** — списание при заказе выбирает партию с ближайшим
  MHD (врезка в атомик заказа — под адверсариальным ревью + паритет-замки).
- **Архитектура: Вариант A** (счётчик=итого, партии=разбивка+реконсиляция) — по умолчанию.
- **Опциональность: тумблер `site_config["lots_enabled"]`** — партии/MHD показываем еде;
  у товаров без партий поведение остатка не меняется (чистый счётчик, как сегодня).

## 4. Развилки (изначальный чеклист — закрыты §4а)

   при заказе выбирает партию с ближайшим MHD, врезка в атомик заказа)? Рекомендую
   стартовать с **MHD-lite**, FEFO — отдельной фазой после проверки на демо.
4. **Опциональность:** партии/MHD включаются пер-тенант (тумблер, показываем еде) или

   захламлять не-еде).

## 5. Фазовая разбивка E1 (Chargen/MHD) — по одному инкременту, миграции помечены

> Каждая фаза = ветка → CI зелёный → чекпоинт → FF-мерж; крупная — под адверсариальным ревью.

- **E1.0 (docs)** — этот план + фиксация решений §4 в task-catalog (ID U-D2W-E1…).
- **E1.1 (⚠️ миграция `inventory/0002`)** — модель `Lot`/`Charge` (TENANT): product/variant,
  `lot_code`, `mhd` (DateField null), `received_at`, `qty_received`, `qty_remaining`,
  `note`. Аддитивно; тумблер `site_config["lots_enabled"]`. Без врезок в движки.
- **E1.2** — приёмка по партиям в кабинете: расширить receipt-форму (Charge + MHD) →
  создаёт `Lot` + пишет `StockMovement(receipt)` в той же atomic (реюз
  `apply_manual_movement`-паттерна). Реконсиляция `Σ lot.qty_remaining ↔ счётчик`.
- **E1.3** — MHD-обзор: секция «Sроки годности» (партии, отсортированы по MHD; бейджи
  «läuft in N Tagen ab» / «abgelaufen»); фильтр/список скоро-истекающих. Beat-алерт
  владельцу (email/Telegram, реюз notifications) — опц.
- **E1.4** — Verderb-по-партии: списание истёкшей партии (kind=adjustment, reason=verderb,
  привязка к Lot) из MHD-обзора одной кнопкой; уменьшает `qty_remaining` + счётчик + леджер.
- **E1.5 (опц., ⚠️ если врезка в атомик)** — FEFO-списание: при продаже/ручном расходе
  выбирать партию с ближайшим MHD. Врезка в orders sale-path под адверсариальным ревью
  (риск: атомик заказа). Только после согласования — может остаться в v2.
- **Демо/тесты:** демо-партии для еды-китов (bakery/butcher) с реалистичными MHD;
  замки на реконсиляцию партий, алерты, обратимость.

## 6. Риски

- **Инвариант D1:** партии — разбивка поверх счётчика (Вариант A), НЕ замена; каждая
  запись партии — в той же atomic, что счётчик; реконсиляция ловит расхождения.
- **FEFO-на-продаже (E1.5)** — единственная врезка в чужой атомик заказа → отдельная
  фаза, адверсариальный ревью-workflow, паритет-замки до правок.
- **Миграции:** каждая — аддитивная/nullable (совместимость со старым кодом, паттерн T5);
  каждая помечается `⚠️ деплой` в build-log/каталоге.
- **Не захламлять не-еду:** тумблер `lots_enabled`; в Простом режиме у не-еды — скрыто.
- **GoBD:** партии/MHD-списания — потенциально Aufbewahrung; леджер уже append-only.
  Полноценный revisionssicher-режим — отметка на будущее, не блокер v1.
